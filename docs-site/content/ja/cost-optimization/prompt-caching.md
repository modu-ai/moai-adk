# プロンプトキャッシング — 損益分岐点分析と実装ガイド

プロンプトキャッシングは、複数のリクエスト全体で同一のプロンプトプレフィックスを再利用することで、90%割引（基本コストの0.1倍）でコスト削減します。このガイドでは、損益分岐点ルール、キャッシュメカニズム、およびMoAIプロジェクトでキャッシングをいつ有効にするかを説明します。

## 損益分岐点ルール

**1時間キャッシュは、セッションで2つ以上の連続するAPIリクエストが発生する場合にのみ有効にしてください。**

単一リクエストセッション（ワンショットクエリ、シングルターンコマンド）は2倍の書き込みプレミアムが発生し、キャッシュの再利用の利点がないため、リクエストあたりのコストはキャッシングなしのベースラインより高くなります。1時間以内に複数ターンの相互作用または反復的なSPEC分析がある場合、キャッシングは2番目のリクエストでコストを相殺し、後続のリクエストで67%以上削減します。

### コスト比較

Claude Opus 4.5を参考にしています：

| シナリオ | キャッシュなし | 1時間キャッシュ付き | 削減 |
|---------|--------|----------|--------|
| 1個リクエスト、10Kトークン | $0.05 | $0.0625 | -25%（プレミアム） |
| 2個リクエスト、10K + 10K | $0.10 | $0.0625 + $0.005 = $0.0675 | 32%削減 |
| 3個リクエスト、10K + 10K + 10K | $0.15 | $0.0625 + 2×$0.005 = $0.0725 | 52%削減 |
| 5個リクエスト、10K + 10K + 10K + 10K + 10K | $0.25 | $0.0625 + 4×$0.005 = $0.0825 | 67%削減 |

損益分岐点は**2個リクエスト**です：最初のリクエストの2倍の書き込みプレミアムは、1時間のTTLウィンドウ内の後続リクエストの90%割引キャッシュ読み取りによって相殺されます。なぜ損益分岐点が重要かというと、プロンプトキャッシングのコスト構造は独特だからです。キャッシュ書き込みには基本コストの2倍がかかりますが、キャッシュ読み取りはわずか0.1倍です。したがって、最初のリクエストで追加コストを支払いますが、2番目のリクエスト以降で大幅な節約が得られます。これは、マルチターンの会話、反復的な分析、または同じコンテキストでの複数のクエリに特に有益です。

## キャッシュ制御の動作原理

プロンプトプレフィックスでキャッシュ制御を有効にすると、キャッシュライフサイクルは次のパターンに従います：

1. **最初のリクエスト（キャッシュ書き込み）**: APIレスポンスが完了した後、プレフィックスはキャッシュに書き込まれます。コスト：`プレフィックス_トークン × 2.0（1時間キャッシュ）または1.25（5分キャッシュ）`。
2. **後続リクエスト（キャッシュ読み取り）**: プレフィックスが同一でTTL内にある場合、キャッシュから取得します。コスト：`プレフィックス_トークン × 0.1`。
3. **自動ルックバック**: システムは最後の20メッセージ内のキャッシュされたプレフィックスマッチをチェックします。見つかった場合は読み取りコストが適用されます。

### キャッシュ配置のベストプラクティス

キャッシュ制御ブレークポイントを**リクエスト別データ前の最後の安定ブロック**に配置してください。安定したシステムプロンプトをキャッシュし、ユーザークエリや動的コンテンツはキャッシュ外に保つことが重要です。

```python
# 正しい：安定したシステムプロンプト（キャッシュ可能）
response = client.messages.create(
    model="claude-opus-4-8",
    max_tokens=1024,
    system=[
        {
            "type": "text",
            "text": "あなたはコードレビュアーです。以下のベストプラクティスに従ってください...",
            "cache_control": {"type": "ephemeral", "ttl": "1h"}
        }
    ],
    # 下記はリクエスト別の変更可能なデータ（キャッシュなし）
    messages=[{"role": "user", "content": user_query}]
)

# 間違い：キャッシュブレークポイントが変更データ上にある
# 現在の時刻：{timestamp}
cache_control={"type": "ephemeral"}
# ^ タイムスタンプがリクエストごとに変わるため、決してマッチしません
```

## 設定：session_ttlとspec_ttl

MoAIキャッシングは`.moai/config/sections/cache.yaml`で設定されます：

```yaml
cache:
  enabled: false  # キャッシングを有効にするにはtrueに設定
  session_ttl: "1h"  # セッションレベルキャッシュTTL
  spec_ttl: "5m"     # SPECボディキャッシュTTL
  min_cacheable_tokens: 2048  # キャッシュ書き込み最小トークン
```

### session_ttl: "off"によるオプトアウト

特定のセッションでキャッシングを無効にするには（例：ワンショットリクエストが支配的な場合）：

```yaml
cache:
  enabled: true
  session_ttl: "off"  # このセッションではキャッシュを無効化
  spec_ttl: "5m"      # SPECボディのみキャッシュを使用
```

`session_ttl: "off"`の場合、セッションレベルのキャッシュ書き込みがスキップされますが、`spec_ttl`が設定されていればSPECボディキャッシュは適用され続けます。これはワンショットリクエストが一般的な割り込み駆動ワークフローに有用です。

## キャッシュパフォーマンスの監視

`moai doctor`を使用してキャッシュヒット率を表示し、キャッシングを有効にするかどうかを決定してください：

```bash
moai doctor --cache-metrics
```

出力例：

```
キャッシュパフォーマンス（過去7日間）：
  キャッシュヒット率：67%
  総キャッシュ読み取り：450Kトークン
  総キャッシュ書き込み：150Kトークン
  削減：$2.15（キャッシング無しと比較で68%コスト削減）
  
シングルターンリクエスト率：12%  ⚠️ 警告：リクエストの12%がシングルターン
                                    （これらはキャッシュの利点なし）。
```

### メトリクスの解釈

- **ヒット率 > 60%**: キャッシュが効果的です。有効を維持してください。
- **ヒット率 30-60%**: 中程度の利点があります。セッションがマルチターン中心であれば有効化を検討してください。
- **シングルターン率 > 30%**: キャッシングの利点は限定的です。2個以上のリクエスト仮定が保たれているか確認してください。
- **最小トークン閾値警告**: `min_cacheable_tokens`を設定して小さなプロンプトのキャッシングを避けてください（オーバーヘッド > 削減）。

## キャッシュミスが発生する場合

キャッシュヒットには以下が必要です：

- ✓ ブレークポイントまで同一のプロンプトプレフィックス
- ✓ TTLウィンドウ内（1時間または5分）
- ✓ 同じワークスペース/組織コンテキスト
- ✓ ブレークポイント前のすべてのブロック変更なし（ツール、システム、トップレベルパラメータ）

キャッシュミス（一般的な原因）：

- ✗ ツール定義変更（ツールパラメータが異なる）
- ✗ Webサーチのオン/オフ切り替え
- ✗ プレフィックスで画像が追加または削除
- ✗ 拡張思考設定の変更
- ✗ ブレークポイント前のコンテンツが異なる（空白含む）

## 最小トークン閾値

キャッシュ書き込みは、プレフィックスが言語モデル固有の最小値を超える場合にのみ発行されます：

- **Claude Opus 4.5, 4.7, 4.8, Haiku 4.5**: 2,048トークン最小
- **Claude Opus 4.1、Sonnetモデル、他のHaikuバージョン**: 1,024トークン最小

最小値未満のリクエストはキャッシングなしで処理されます（エラーなし — 自動フォールバック）。

## プレウォーミング（高度）

最初のユーザー相互作用のキャッシュミスレイテンシを排除するには：

```python
# キャッシュプレウォーミング（ユーザー到着前）
client.messages.create(
    model="claude-opus-4-8",
    max_tokens=0,  # 出力トークン課金なし
    system="長いシステムプロンプト（5000トークン）...",
    cache_control={"type": "ephemeral", "ttl": "1h"},
    messages=[{"role": "user", "content": "warmup"}]
)
# コスト：system_tokens × $2.0/MTok（キャッシュ書き込み）

# 後：ユーザーリクエストがウォームキャッシュをヒット
client.messages.create(
    model="claude-opus-4-8",
    max_tokens=1024,
    system="長いシステムプロンプト（同一）...",
    cache_control={"type": "ephemeral", "ttl": "1h"},
    messages=[{"role": "user", "content": user_input}]
)
# コスト：system_tokens × $0.1/MTok（ウォーミングからのキャッシュ読み取り）
```

## まとめ

- **キャッシュを有効化**: 1時間内に2個以上の連続するAPIリクエストがあるセッション
- **キャッシュを無効化**: ワンショットクエリまたは割り込み駆動ワークフロー
- **監視**: `moai doctor --cache-metrics`を使用して実際のヒット率を測定
- **最適化**: キャッシュブレークポイントを安定したコンテンツ（システムプロンプト、指示）に配置し、変更データ（クエリ、タイムスタンプ）には配置しないでください

詳細については、[Anthropicプロンプトキャッシングドキュメント](https://platform.claude.com/docs/en/docs/build-with-claude/prompt-caching)を参照してください。
