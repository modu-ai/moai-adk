# 프롬프트 캐싱 — 손익분기 분석 및 구현 가이드

프롬프트 캐싱은 동일한 프롬프트 접두사를 여러 요청에서 재사용하여 90% 할인된 비용으로 추론 비용을 절감합니다(기본 비용의 0.1배). 이 가이드는 손익분기 규칙, 캐시 메커니즘, 그리고 MoAI 프로젝트에서 캐싱을 언제 활성화할지 설명합니다.

## 손익분기 규칙

**1시간 캐시는 세션에서 2개 이상의 연속 API 요청이 발생할 때만 활성화하세요.**

단일 요청 세션(원샷 쿼리, 단일 턴 명령)은 2배의 기록 프리미엄이 발생하며 캐시 재사용 이득이 없어서 요청당 비용이 캐싱 없는 기본선보다 높습니다. 1시간 이내에 여러 턴 상호작용이나 반복 SPEC 분석이 있다면 캐싱은 두 번째 요청에서 비용을 상쇄하고 후속 요청에서 67% 이상을 절감합니다.

### 비용 비교

Claude Opus 4.5를 기준으로:

| 시나리오 | 캐싱 없음 | 1시간 캐시 포함 | 절감 |
|---------|----------|-----------------|-------|
| 1개 요청, 10K 토큰 | $0.05 | $0.0625 | -25% (프리미엄) |
| 2개 요청, 10K + 10K | $0.10 | $0.0625 + $0.005 = $0.0675 | 32% 절감 |
| 3개 요청, 10K + 10K + 10K | $0.15 | $0.0625 + 2×$0.005 = $0.0725 | 52% 절감 |
| 5개 요청, 10K + 10K + 10K + 10K + 10K | $0.25 | $0.0625 + 4×$0.005 = $0.0825 | 67% 절감 |

손익분기점은 **2개 요청**입니다: 첫 번째 요청의 2배 기록 프리미엄은 1시간 TTL 윈도우 내 후속 요청의 90% 할인 캐시 읽기로 회수됩니다.

## 캐시 제어 작동 원리

프롬프트 접두사에서 캐시 제어를 활성화하면 캐시 라이프사이클은 다음 패턴을 따릅니다:

1. **첫 번째 요청(캐시 기록)**: API 응답이 완료된 후 접두사가 캐시에 기록됩니다. 비용: `접두사_토큰 × 2.0 (1시간 캐시) 또는 1.25 (5분 캐시)`.
2. **후속 요청(캐시 읽기)**: 접두사가 동일하고 TTL 내에 있으면 캐시에서 검색합니다. 비용: `접두사_토큰 × 0.1`.
3. **자동 역추적**: 시스템은 마지막 20개 메시지에서 캐시된 접두사 일치를 확인합니다. 발견되면 읽기 비용이 적용됩니다.

### 캐시 배치 모범 사례

캐시 제어 중단점을 **요청별 데이터 앞의 마지막 안정적 블록**에 배치하세요:

```python
# 올바름: 안정적 시스템 프롬프트(캐시 가능)
response = client.messages.create(
    model="claude-opus-4-8",
    max_tokens=1024,
    system=[
        {
            "type": "text",
            "text": "당신은 코드 리뷰어입니다...",
            "cache_control": {"type": "ephemeral", "ttl": "1h"}
        }
    ],
    # 아래 변경 가능한 요청별 데이터(캐시 안 함)
    messages=[{"role": "user", "content": user_query}]
)

# 잘못됨: 캐시 중단점이 변경 데이터에 있음
# 현재 시간: {timestamp}
cache_control={"type": "ephemeral"}
# ^ 타임스탬프가 매 요청마다 변하므로 절대 일치하지 않음
```

## 설정: session_ttl과 spec_ttl

MoAI 캐싱은 `.moai/config/sections/cache.yaml`에서 설정됩니다:

```yaml
cache:
  enabled: false  # 캐싱을 활성화하려면 true로 설정
  session_ttl: "1h"  # 세션 수준 캐시 TTL
  spec_ttl: "5m"     # SPEC 본문 캐시 TTL
  min_cacheable_tokens: 2048  # 캐시 기록 최소 토큰
```

### session_ttl: "off"를 통한 선택 해제

특정 세션에서 캐싱을 비활성화하려면(예: 원샷 요청이 지배적일 때):

```yaml
cache:
  enabled: true
  session_ttl: "off"  # 이 세션에서 캐시 비활성화
  spec_ttl: "5m"      # SPEC 본문만 캐시 사용
```

`session_ttl: "off"`일 때:
- 세션 수준 캐시 기록이 건너뜀
- `spec_ttl`이 구성되면 SPEC 본문 캐시는 계속 적용
- 단일 요청이 일반적인 중단 기반 워크플로우에 유용

## 캐시 성능 모니터링

`moai doctor`를 사용하여 캐시 히트율을 보고 캐싱 활성화 여부를 결정하세요:

```bash
moai doctor --cache-metrics
```

출력 예시:

```
캐시 성능 (지난 7일):
  캐시 히트율: 67%
  총 캐시 읽기: 450K 토큰
  총 캐시 기록: 150K 토큰
  절감: $2.15 (캐싱 없는 경우 대비 68% 비용 절감)
  
단일 턴 요청 비율: 12%  ⚠️ 경고: 요청의 12%가 단일 턴
                            (이들에 대한 캐시 이득 없음).
```

### 메트릭 해석

- **히트율 > 60%**: 캐시가 효과적입니다. 활성화 유지하세요.
- **히트율 30-60%**: 중간 정도의 이득. 세션이 다중 턴 중심이면 활성화 고려하세요.
- **단일 턴 비율 > 30%**: 캐싱의 이득이 제한적입니다. 2개 이상 요청 가정이 유지되는지 확인하세요.
- **최소 토큰 임계값 경고**: `min_cacheable_tokens`를 구성하여 작은 프롬프트 캐싱을 피하세요(오버헤드 > 절감).

## 캐시 미스가 발생하는 경우

캐시 히트에는 다음이 필요합니다:

- ✓ 중단점까지 동일한 프롬프트 접두사
- ✓ TTL 윈도우 내(1시간 또는 5분)
- ✓ 동일한 워크스페이스/조직 컨텍스트
- ✓ 중단점 앞의 모든 블록 변경 없음(도구, 시스템, 최상위 레벨 매개변수)

캐시 미스(일반적 원인):

- ✗ 도구 정의 변경(도구 매개변수 다름)
- ✗ 웹 검색 토글 켜짐/꺼짐
- ✗ 접두사에 이미지 추가 또는 제거
- ✗ 확장 사고 설정 변경
- ✗ 중단점 앞 콘텐츠 다름(공백 포함)

## 최소 토큰 임계값

캐시 기록은 접두사가 언어 모델별 최소값을 초과할 때만 발행됩니다:

- **Claude Opus 4.5, 4.7, 4.8, Haiku 4.5**: 2,048 토큰 최소
- **Claude Opus 4.1, Sonnet 모델, 기타 Haiku 버전**: 1,024 토큰 최소

최소값 미만의 요청은 캐싱 없이 처리됩니다(오류 없음 — 자동 폴백).

## 사전 워밍(고급)

첫 사용자 상호작용의 캐시 미스 레이턴시를 제거하려면:

```python
# 캐시 사전 워밍(사용자 도착 전)
client.messages.create(
    model="claude-opus-4-8",
    max_tokens=0,  # 출력 토큰 청구 없음
    system="긴 시스템 프롬프트(5000 토큰)...",
    cache_control={"type": "ephemeral", "ttl": "1h"},
    messages=[{"role": "user", "content": "warmup"}]
)
# 비용: system_tokens × $2.0/MTok (캐시 기록)

# 나중: 사용자 요청이 워밍된 캐시 히트
client.messages.create(
    model="claude-opus-4-8",
    max_tokens=1024,
    system="긴 시스템 프롬프트(동일)...",
    cache_control={"type": "ephemeral", "ttl": "1h"},
    messages=[{"role": "user", "content": user_input}]
)
# 비용: system_tokens × $0.1/MTok (워밍에서 캐시 읽기)
```

## 요약

- **캐시 활성화**: 1시간 내 2개 이상의 연속 API 요청이 있는 세션
- **캐시 비활성화**: 원샷 쿼리 또는 중단 기반 워크플로우
- **모니터링**: `moai doctor --cache-metrics`를 사용하여 실제 히트율 측정
- **최적화**: 캐시 중단점을 안정적 콘텐츠(시스템 프롬프트, 지시)에 배치하고 변경 데이터(쿼리, 타임스탐프)에는 배치하지 마세요

자세한 내용은 [Anthropic 프롬프트 캐싱 문서](https://platform.claude.com/docs/en/docs/build-with-claude/prompt-caching)를 참조하세요.
