# 提示词缓存 — 损益平衡点分析和实现指南

提示词缓存通过在多个请求中重用相同的提示词前缀，以90%折扣（基础成本的0.1倍）来降低推理成本。本指南解释了损益平衡点规则、缓存机制，以及何时为MoAI项目启用缓存。

## 损益平衡点规则

**仅当您的会话在1小时内产生2个或更多连续的API请求时，才启用1小时缓存。**

单个请求会话（一次性查询、单轮命令）会产生2倍的写入溢价，且没有缓存重用的好处——每个请求的成本高于无缓存基线。对于多轮交互或1小时内重复的SPEC分析，缓存在第二个请求上就能收回成本，并在后续请求上节省67%以上。

### 成本对比

使用Claude Opus 4.5作为参考：

| 场景 | 无缓存 | 带1小时缓存 | 节省 |
|------|-------|----------|------|
| 1个请求，10K令牌 | $0.05 | $0.0625 | -25%（溢价） |
| 2个请求，10K + 10K | $0.10 | $0.0625 + $0.005 = $0.0675 | 节省32% |
| 3个请求，10K + 10K + 10K | $0.15 | $0.0625 + 2×$0.005 = $0.0725 | 节省52% |
| 5个请求，10K + 10K + 10K + 10K + 10K | $0.25 | $0.0625 + 4×$0.005 = $0.0825 | 节省67% |

损益平衡点在**2个请求**处：第一个请求的2倍写入溢价由1小时TTL窗口内后续请求的90%折扣缓存读取来抵消。损益平衡点之所以重要，是因为提示词缓存的成本结构是独特的。缓存写入成本为基础成本的2倍，但缓存读取仅为0.1倍。因此，您在第一个请求上支付额外费用，但从第二个请求开始获得大幅节省。这对多轮对话、迭代分析或在相同上下文中进行多个查询特别有益。

## 缓存控制如何工作

在提示词前缀上启用缓存控制时，缓存生命周期遵循以下模式：

1. **首次请求（缓存写入）**：API响应完成后，前缀被写入缓存。成本：`前缀_令牌 × 2.0（1小时缓存）或1.25（5分钟缓存）`。
2. **后续请求（缓存读取）**：如果前缀相同且在TTL内，系统会从缓存检索。成本：`前缀_令牌 × 0.1`。
3. **自动回溯**：系统检查最后20条消息中是否有缓存的前缀匹配。如果找到，应用读取成本。

### 缓存放置最佳实践

将缓存控制断点放在**每个请求数据之前的最后稳定块**上。缓存稳定的系统提示词很重要，同时将用户查询和动态内容保留在缓存之外。

```python
# 正确：稳定的系统提示词（可缓存）
response = client.messages.create(
    model="claude-opus-4-8",
    max_tokens=1024,
    system=[
        {
            "type": "text",
            "text": "你是一个代码审查员。遵循以下最佳实践...",
            "cache_control": {"type": "ephemeral", "ttl": "1h"}
        }
    ],
    # 下面是每个请求可变的数据（不缓存）
    messages=[{"role": "user", "content": user_query}]
)

# 错误：缓存断点在变化数据上
# 当前时间：{timestamp}
cache_control={"type": "ephemeral"}
# ^ 这永远不会匹配，因为时间戳每个请求都会变化
```

## 配置：session_ttl和spec_ttl

MoAI缓存在`.moai/config/sections/cache.yaml`中配置：

```yaml
cache:
  enabled: false  # 设置为true来启用缓存
  session_ttl: "1h"  # 会话级别缓存TTL
  spec_ttl: "5m"     # SPEC正文缓存TTL
  min_cacheable_tokens: 2048  # 缓存写入最小令牌
```

### 通过session_ttl: "off"选择退出

要在特定会话中禁用缓存（例如，当一次性请求占主导时）：

```yaml
cache:
  enabled: true
  session_ttl: "off"  # 禁用此会话的缓存
  spec_ttl: "5m"      # 仅SPEC正文使用缓存
```

当`session_ttl: "off"`时，会话级别的缓存写入被跳过，但如果配置了`spec_ttl`，SPEC正文缓存仍然适用。这对于中断驱动的工作流（其中单个请求很常见）很有用。

## 监视缓存性能

使用`moai doctor`查看缓存命中率并决定是否启用缓存：

```bash
moai doctor --cache-metrics
```

输出示例：

```
缓存性能（最近7天）：
  缓存命中率：67%
  总缓存读取：450K令牌
  总缓存写入：150K令牌
  节省：$2.15（与无缓存相比节省68%成本）
  
单轮请求比例：12%  ⚠️ 警告：12%的请求是单轮
                        （这些没有缓存好处）。
```

### 解释指标

- **命中率 > 60%**：缓存有效。保持启用。
- **命中率 30-60%**：适度的好处。如果会话是多轮主导的，考虑启用。
- **单轮请求比例 > 30%**：缓存的好处有限。验证2个以上请求的假设是否成立。
- **最小令牌阈值警告**：配置`min_cacheable_tokens`来避免缓存小提示词（开销>节省）。

## 何时发生缓存未命中

缓存命中需要：

- ✓ 到断点为止的相同提示词前缀
- ✓ 在TTL窗口内（1小时或5分钟）
- ✓ 相同的工作区/组织背景
- ✓ 断点前的所有块未变化（工具、系统、顶级参数）

缓存未命中（常见原因）：

- ✗ 工具定义变化（工具参数不同）
- ✗ 网络搜索已切换
- ✗ 在前缀中添加或移除了图像
- ✗ 扩展思考设置变化
- ✗ 断点前的内容不同（包括空白）

## 最小令牌阈值

仅当前缀超过特定于语言模型的最小值时才发出缓存写入：

- **Claude Opus 4.5、4.7、4.8、Haiku 4.5**：2,048令牌最小值
- **Claude Opus 4.1、Sonnet模型、其他Haiku版本**：1,024令牌最小值

低于最小值的请求不使用缓存处理（无错误——自动回退）。

## 预热（高级）

要消除首次用户交互的缓存未命中延迟：

```python
# 预热缓存（用户到达前）
client.messages.create(
    model="claude-opus-4-8",
    max_tokens=0,  # 无输出令牌计费
    system="长系统提示词（5000令牌）...",
    cache_control={"type": "ephemeral", "ttl": "1h"},
    messages=[{"role": "user", "content": "warmup"}]
)
# 成本：system_tokens × $2.0/MTok（缓存写入）

# 后来：用户请求命中预热缓存
client.messages.create(
    model="claude-opus-4-8",
    max_tokens=1024,
    system="长系统提示词（相同）...",
    cache_control={"type": "ephemeral", "ttl": "1h"},
    messages=[{"role": "user", "content": user_input}]
)
# 成本：system_tokens × $0.1/MTok（来自预热的缓存读取）
```

## 总结

- **启用缓存**：用于在1小时内产生2个或更多连续API请求的会话
- **禁用缓存**：用于一次性查询或中断驱动的工作流
- **监视**：使用`moai doctor --cache-metrics`测量实际命中率
- **优化**：将缓存断点放在稳定的内容（系统提示词、说明）上，不要放在变化的数据（查询、时间戳）上

更多详情，请参阅[Anthropic提示词缓存文档](https://platform.claude.com/docs/en/docs/build-with-claude/prompt-caching)。
