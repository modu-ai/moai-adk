# MoAI-ADK 개요

## 핵심 정보

**MoAI-ADK** (Modu-AI's Agentic Development Kit)는 SPEC-First TDD 개발 방식을 Python으로 구현한 프레임워크입니다.

| 항목 | 정보 |
|------|------|
| **구성** | 57개 Python 모듈 |
| **코드량** | 10,006 줄 |
| **핵심 계층** | 8개 주요 서브시스템 |
| **지원 언어** | 20개 프로그래밍 언어 |
| **버전** | v1.0.0 |

---

## 아키텍처 개요

MoAI-ADK는 **4계층 레이어드 아키텍처**로 설계되었습니다:

```
┌─────────────────────────────────────┐
│   Layer 1: CLI Interface            │  (사용자 인터페이스)
│   - init, doctor, status, update    │
│   - Rich console 출력               │
└─────────────────────────────────────┘
              ↓
┌─────────────────────────────────────┐
│   Layer 2: Business Logic (Core)    │  (비즈니스 로직)
│   - Project Management              │
│   - Git Operations                  │
│   - TAG System                      │
│   - Quality Assurance (TRUST 5)     │
│   - Template Processing             │
└─────────────────────────────────────┘
              ↓
┌─────────────────────────────────────┐
│   Layer 3: Integration & Utils      │  (통합/유틸)
│   - Configuration Management        │
│   - Logging (민감 정보 마스킹)       │
│   - System Diagnostics              │
└─────────────────────────────────────┘
              ↓
┌─────────────────────────────────────┐
│   Layer 4: Templates & Plugins      │  (템플릿/플러그인)
│   - Claude Integration              │
│   - Project Templates               │
│   - Marketplace Plugins             │
└─────────────────────────────────────┘
```

### 8개 주요 서브시스템

| 서브시스템 | 책임 | 파일 수 |
|----------|------|--------|
| **CLI** | 사용자 인터페이스 (init, doctor, status, update) | 7개 |
| **Project** | 프로젝트 초기화, 5단계 설정 | 6개 |

## SPEC-First TDD 워크플로우

### ① SPEC 작성
`.moai/specs/SPEC-001/spec.md`에 요구사항을 EARS 포맷으로 정의합니다:

```markdown
## Requirements (요구사항)

### UBIQUITOUS (보편적)
- REQ-001: 시스템은 사용자 입력을 검증해야 한다

### EVENT-DRIVEN (이벤트 기반)
- REQ-002: WHEN 사용자가 제출하면, 시스템은 검증해야 한다

### STATE-DRIVEN (상태 기반)
- REQ-003: WHILE 처리 중일 때, 시스템은 상태를 유지해야 한다
```

### ② TEST 작성
`tests/test_feature.py`에서 테스트를 먼저 작성합니다 (RED):

```python
# 기능 테스트 작성
# tests/test_feature.py
def test_feature_success():
    result = feature()
    assert result == expected
```

### ③ CODE 구현
`src/feature.py`에서 코드를 작성합니다 (GREEN):

```python
# 기능 구현
# src/feature.py
def feature():
    """기능 설명"""
    return expected
```

### ④ 문서 작성
`.moai/docs/feature.md`에서 문서를 작성합니다 (REFACTOR):

```markdown
# 기능 설명

이 기능은 X를 수행합니다.
```

### ⑤ TAG 검증
`moai validate`로 SPEC→CODE→TEST→DOC 연결을 자동 확인합니다:

```bash
moai validate
# ✓ SPEC → CODE 연결됨
# ✓ CODE → TEST 연결됨
# ✓ TEST → DOC 연결됨
```

## 핵심 특징

### ✅ SPEC-First
- 요구사항을 먼저 문서화
- EARS 포맷으로 5가지 요구사항 타입 지원
- AI와 함께 명확한 의도 정의

### ✅ TDD 자동화
- RED → GREEN → REFACTOR 추적
- 테스트 커버리지 검증 (≥85%)
- 코드 리뷰 자동화

### ✅ TAG 시스템
- `@SPEC:*`, `@CODE:*`, `@TEST:*`, `@DOC:*` 체인
- 중복 감지, 고아 감지, 연결 검증
- 다중 형식 보고서 (JSON, Markdown, Plain Text)

### ✅ 품질 관리 (TRUST 5)
- **T**est: 테스트 커버리지 ≥85%
- **R**eadable: 파일 300줄, 함수 50줄 이하
- **U**nified: 타입 안전성 검증
- **S**ecured: 보안 스캔
- **T**rackable: TAG 체인 무결성

### ✅ Git 자동화
- 기능 브랜치 자동 생성
- 커밋 메시지 포맷팅
- Pre-commit 훅 통합
- 체크포인트 시스템 (작업 스냅샷)

### ✅ 20개 언어 지원
Python, JavaScript, TypeScript, Go, Rust, Java, C++, C#, Ruby, PHP, Kotlin, Swift, Dart, R, Scala, SQL, Shell, C, 그 외 다수

## 어떤 팀을 위한가?

### ✅ 추천하는 경우
- AI를 도움받지만 신뢰성이 필요한 팀
- SPEC 기반 요구사항을 관리하는 팀
- 코드-테스트-문서 일관성이 중요한 팀
- Python 기반 프로젝트

### ⚠️ 고려가 필요한 경우
- 매우 작은 스크립트 프로젝트 (학습 곡선 있음)
- 비Python 기반 프로젝트 (확장 필요)
- 이미 확고한 개발 방식이 있는 팀

## 주요 특징

### 1. SPEC-First 접근

모든 개발은 명확한 요구사항 정의(SPEC)부터 시작합니다.

- **EARS 문법** 사용: WHEN-SHALL 형식의 명확한 요구사항
- **@TAG 시스템**: 모든 요구사항에 고유 ID 부여
- **추적 가능**: SPEC → TEST → CODE 연결 보장

### 2. Alfred SuperAgent

19개 AI 에이전트 팀이 협력하여 개발 전 과정을 지원합니다.

- **10개 핵심 에이전트**: spec-builder, tdd-implementer, doc-syncer 등
- **6개 전문 에이전트**: 언어별, 도메인별 전문가
- **55개 Claude Skills**: 재사용 가능한 지식 캡슐

### 3. TDD 자동화

Test-Driven Development를 자동으로 수행합니다.

- **RED**: 실패하는 테스트 작성
- **GREEN**: 테스트를 통과하는 최소 구현
- **REFACTOR**: 코드 품질 개선

### 4. 문서 동기화

코드와 문서가 항상 일치합니다.

- **자동 업데이트**: `/alfred:3-sync` 명령 하나로 모든 문서 동기화
- **Living Documents**: 코드 변경에 따라 자동 갱신
- **일관성 보장**: README, CHANGELOG, API 문서 모두 동기화

## 왜 MoAI-ADK인가?

### 기존 AI 개발의 문제점

| 문제 | 기존 방식 | MoAI-ADK 해결책 |
|------|-----------|----------------|
| 요구사항 불명확 | 구두로 설명, 의도 불일치 | SPEC 문서로 명확화 |
| 테스트 부재 | "나중에 테스트" → 영원히 안 함 | TDD 강제, 85%+ 커버리지 |
| 문서 낙후 | 코드 수정 후 문서 업데이트 누락 | 자동 동기화 |
| 컨텍스트 손실 | 매번 처음부터 설명 | Alfred가 프로젝트 이력 기억 |

### MoAI-ADK의 장점

1. **신뢰할 수 있는 코드**: SPEC 기반 테스트로 검증된 코드
2. **빠른 개발**: 명확한 SPEC으로 시행착오 감소
3. **낮은 유지보수 비용**: 문서와 코드 일치로 이해 쉬움
4. **효율적 협업**: SPEC과 TAG로 명확한 커뮤니케이션
