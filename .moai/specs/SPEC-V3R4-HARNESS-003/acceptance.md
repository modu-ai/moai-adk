### AC-HRN-CLS-001 — Stage-1 Backward Compatibility (Stage 2 OFF)

**Linked REQs**: REQ-HRN-CLS-001, REQ-HRN-CLS-004

**Given** a project with `.moai/config/sections/harness.yaml` containing `learning.classifier.stage_2_enabled: false` (the default, equivalent to the key being absent), AND a synthetic fixture log `testdata/stage1_baseline.jsonl` containing 1000 events generated by the existing `learner_test.go:53-74` pattern (10 distinct (event_type, subject, context_hash) tuples each repeated 100 times),

**When** the test invokes `AggregatePatterns("testdata/stage1_baseline.jsonl")`,

**Then** the returned `map[string]*Pattern` MUST be byte-identical to a golden baseline file `testdata/stage1_baseline_patterns.json` captured from the pre-V3R4-003 implementation. Specifically:
- The map MUST contain exactly 10 entries (one per (event_type, subject, context_hash) tuple).
- Each entry's `Key` MUST use the three-field form `"{event_type}:{subject}:{context_hash}"`.
- Each entry's `Count` MUST be 100.
- Each entry's `Confidence` MUST equal the `defaultConfidence` constant (1.0).
- No new entries (Stage 2 merged patterns) MUST appear.
- No JSONL line MUST be appended to `.moai/harness/cluster-merges.jsonl` (the file MUST remain absent if it did not pre-exist). (maps REQ-HRN-CLS-001, REQ-HRN-CLS-004)

**Verification**:
- Test fixture: `internal/harness/classifier_cluster_test.go::TestStage1BackwardCompat_StageDisabled`.
- Setup: write `learning.classifier.stage_2_enabled: false` to harness.yaml; pre-generate `testdata/stage1_baseline.jsonl` and `testdata/stage1_baseline_patterns.json` (golden).
- Assert: `reflect.DeepEqual(actual, golden)` returns true; `os.Stat(".moai/harness/cluster-merges.jsonl")` returns `IsNotExist`.

### AC-HRN-CLS-002 — Stage 2 ON: 10-Singleton Fixture Aggregates to 1 Merged Pattern

**Linked REQs**: REQ-HRN-CLS-002, REQ-HRN-CLS-003

**Given** a project with `learning.classifier.stage_2_enabled: true`, `learning.classifier.hamming_threshold: 3`, `learning.classifier.cluster_min_size: 3`, `learning.classifier.feature_fields: ["subject", "prompt_preview", "prompt_lang", "agent_name", "agent_type"]`, AND a synthetic fixture `testdata/stage2_similar_10.jsonl` containing exactly 10 singleton events ALL with `event_type: "user_prompt"` AND subjects that vary by SPEC ID (e.g., `/moai plan SPEC-001`, `/moai plan SPEC-002`, …, `/moai plan SPEC-010`) AND identical `prompt_lang: "en"` AND — critically — IDENTICAL `prompt_preview` content across all 10 events (the realistic Tier-2 scenario: a user submitting the same intent prompt across multiple sessions). The shared `prompt_preview` is a 64-byte string such as `"plan a new authentication SPEC with EARS requirements..."`. The dominant SimHash feature is the shared `prompt_preview`, so pairwise Hamming distance between any two events is dominated by tiny subject-suffix variation only,

**When** the test invokes `AggregatePatterns("testdata/stage2_similar_10.jsonl")`,

**Then** the returned pattern map MUST contain exactly 11 entries: 10 Stage-1 singleton records (each unchanged, Count=1) AND 1 Stage-2 merged Pattern record (synthetic key with Count=10 and aggregated Confidence). The merged Pattern MUST have:
  - `Key`: a two-field canonical form `"user_prompt:<lexicographically-smallest-subject>"` (e.g., `"user_prompt:/moai plan SPEC-001"`)
  - `EventType`: `EventTypeUserPrompt`
  - `Subject`: matches the canonical form
  - `ContextHash`: empty string `""`
  - `Count`: 10 (sum of the 10 singleton counts)
  - `Confidence`: equal to `defaultConfidence` (1.0, the arithmetic mean of 10 identical 1.0 values)
- The pairwise Hamming distance between any two cluster members MUST be ≤ 3 (asserted via direct SimHash computation in the test).
- The merged Pattern's `Tier` after `ClassifyTier` invocation with default `tier_thresholds: [1, 3, 5, 10]` MUST equal `TierAutoUpdate` (since Count=10 ≥ threshold[3]=10). (maps REQ-HRN-CLS-002, REQ-HRN-CLS-003)

**Verification**:
- Test fixture: `internal/harness/classifier_cluster_test.go::TestStage2Aggregates10SimilarSingletons`.
- Setup: write Stage 2 config to harness.yaml (`hamming_threshold: 3`); generate 10 events sharing one identical 64-byte `prompt_preview` value, varying only subject SPEC ID suffix and `prompt_lang: "en"` constant.
- Assert (a): pattern map contains exactly 11 entries; the merged record matches expected Key, Count=10, Confidence=1.0.
- Assert (b): pairwise Hamming distance between any two cluster members MUST be ≤ 3 (asserted via direct SimHash computation: iterate over `C(10,2) = 45` pairs and confirm each pair's Hamming ≤ 3).
- Assert (c): `ClassifyTier(merged, [1,3,5,10]) == TierAutoUpdate`.

### AC-HRN-CLS-004 — Cluster Merge Emission: Pattern Record Shape

**Linked REQs**: REQ-HRN-CLS-005, REQ-HRN-CLS-006

**Given** a Stage-2 enabled run that has identified one cluster of 5 singletons (sufficient for `cluster_min_size: 3`), where the 5 members have Confidence values `[1.0, 1.0, 0.9, 0.8, 1.0]`,

**When** the cluster merge step emits the synthetic Pattern,

**Then** the emitted Pattern MUST have:
- `Key`: equal to `"{event_type}:{lex_min_subject}"` (deterministic — lexicographically-smallest subject from the cluster members)
- `EventType`: the shared event type of all 5 members (Stage-2 clusters only contain singletons sharing the same first key field)
- `Subject`: the lexicographically-smallest subject from the cluster
- `ContextHash`: empty string `""` (Stage-2 merged patterns are session-agnostic)
- `Count`: 5 (sum)
- `Confidence`: 0.94 (the arithmetic mean: (1.0+1.0+0.9+0.8+1.0)/5 = 0.94)
- `Tier`: result of `ClassifyTier(merged, [1,3,5,10])` — for Count=5 and Confidence=0.94, this is `TierRule` (since Count ≥ thresholds[2]=5 and Confidence ≥ 0.70).

**And** the `Tier` enum value MUST be one of the four constants `TierObservation`, `TierHeuristic`, `TierRule`, `TierAutoUpdate` (no new enum values introduced). The `String()` method MUST return `"observation"`, `"heuristic"`, `"rule"`, or `"auto_update"` exactly. (maps REQ-HRN-CLS-005, REQ-HRN-CLS-006)

**Verification**:
- Test fixture: `internal/harness/classifier_cluster_test.go::TestClusterMergeEmissionShape`.
- Synthetic input: 5-member cluster with controlled confidence values.
- Assert: emitted Pattern fields match the expected shape exactly; `Tier == TierRule`; `Tier.String() == "rule"`.

### AC-HRN-CLS-008 — Cluster-Merge Audit Log Entry Schema

**Linked REQs**: REQ-HRN-CLS-013

**Given** a Stage-2 enabled run that emits exactly one cluster merge for a 4-member singleton cluster,

**When** the audit log emission step writes to `.moai/harness/cluster-merges.jsonl`,

**Then** the file MUST contain exactly ONE JSONL line whose JSON object includes:
- `ts`: a string in ISO-8601 UTC format (e.g., `"2026-05-15T14:23:45.123Z"`)
- `member_keys`: a JSON array of 4 strings, each in the Stage-1 three-field form `"{event_type}:{subject}:{context_hash}"`
- `member_counts`: a JSON array of 4 integers, all equal to 1 (singletons)
- `hamming_distances`: a JSON array of 6 integers (the upper-triangle of the 4×4 pairwise distance matrix, in row-major order: d[0,1], d[0,2], d[0,3], d[1,2], d[1,3], d[2,3])
- `merged_key`: a single string in the two-field canonical form `"{event_type}:{subject_canonical}"`
- `merged_count`: integer 4 (sum of members)
- `confidence`: a float between 0.0 and 1.0 (the arithmetic mean of member confidences)

**And** the file MUST use append-only semantics: a second clustering run that adds 2 more merges produces a file with exactly 3 JSONL lines (in append order). (maps REQ-HRN-CLS-013)

**Verification**:
- Test fixture: `internal/harness/classifier_cluster_audit_test.go::TestClusterMergeAuditLogSchema`.
- Setup: synthetic 4-member cluster; run clustering; read JSONL; unmarshal.
- Assert: all required fields present with correct types; `len(hamming_distances) == 6`; `merged_count == 4`.

### AC-HRN-CLS-014 — Invalid Config Fails Safe to Stage-1-Only

**Linked REQs**: REQ-HRN-CLS-018, REQ-HRN-CLS-016

**Given** a project with `learning.classifier.stage_2_enabled: true` AND ONE of the following invalid configurations:
- Sub-case 1: `learning.classifier.hamming_threshold: -5` (negative)
- Sub-case 2: `learning.classifier.hamming_threshold: 99` (exceeds 64-bit space)
- Sub-case 3: `learning.classifier.cluster_min_size: 1` (below floor of 2)
- Sub-case 4: `learning.classifier.similarity_algorithm: "tfidf"` (not in `{simhash, none}`)
- Sub-case 5: `learning.classifier.similarity_algorithm: "garbage_value_xyz"` (unknown)
- Sub-case 6: invalid type — config sets `hamming_threshold: '3.5'` (string) or `hamming_threshold: 3.5` (float). Then `yaml.Unmarshal` returns a type error before `Validate()` runs. The loader MUST catch this error, log a WARN entry, and fall back to defaults (`hamming_threshold: 3`).

**When** the test invokes `AggregatePatterns` with any of these configurations and a fixture that would produce Stage-2 merges under valid config,

**Then** the classifier MUST fall back to Stage-1-only behavior (equivalent to `stage_2_enabled: false`):
- The returned pattern map MUST be byte-identical to the Stage-1 baseline (no merged Pattern records emitted).
- No JSONL entry MUST be appended to `.moai/harness/cluster-merges.jsonl`.
- A single one-line warning MUST be emitted to stderr identifying the invalid key and the fallback action (e.g., `"WARN: learning.classifier.hamming_threshold=-5 out of range [0,64]; Stage 2 disabled."` for in-range validation, or `"WARN: learning.classifier.hamming_threshold yaml type mismatch (expected int); falling back to defaults."` for sub-case 6).

**And** the function MUST return `(patterns, nil)` — NO error MUST be raised to the caller. The fallback is non-blocking. (maps REQ-HRN-CLS-018, REQ-HRN-CLS-016)

**Verification**:
- Table-driven test: `internal/harness/classifier_cluster_test.go::TestInvalidConfigFailsSafeToStage1` iterates over the 6 invalid-config sub-cases.
- Per case: assert returned pattern map == Stage-1 baseline; `os.Stat(cluster-merges.jsonl)` returns `IsNotExist`; stderr capture contains the expected warning substring; function returns nil error.

## Edge Cases

### EDGE-001 — Empty usage-log.jsonl

If `.moai/harness/usage-log.jsonl` is absent or empty, `AggregatePatterns` MUST return an empty pattern map (unchanged behavior from `learner.go:51-53`). Stage 2 MUST short-circuit with an empty input set; no SimHash computation, no audit log emission.

### EDGE-002 — All events are repeated patterns (no singletons)

If the usage-log contains only patterns with Count ≥ 2 (no singletons after Stage 1), Stage 2 MUST short-circuit because there are no singletons to cluster. No merged Pattern emission, no audit log entry.

### EDGE-003 — Single singleton (cluster of size 1)

If exactly 1 singleton exists in Stage 1 output, Stage 2 MUST short-circuit (cluster of size 1 < `cluster_min_size` of 3 minimum, even at the floor of 2). No merge.

### EDGE-004 — Cluster of size 2 below cluster_min_size

If 2 singletons have Hamming distance ≤ threshold but `cluster_min_size: 3`, Stage 2 MUST NOT emit a merge. The 2 singletons remain in the pattern map as their original Stage-1 records.

### EDGE-005 — All singletons in one giant cluster

If 100 singletons all have pairwise Hamming distance ≤ threshold (e.g., a pathological fixture with identical fingerprints), Stage 2 MUST emit ONE merged Pattern with Count=100. The cluster-merges.jsonl entry MUST contain all 100 `member_keys` AND MUST cap `hamming_distances` at exactly 20 elements (the first 20 pair distances in row-major upper-triangle order), MUST set `truncated: true`, and MUST set `hamming_pair_count: 4950` (the full pair count `100*99/2`) per REQ-HRN-CLS-013. This caps the JSONL line size below the 64KB `bufio.Scanner` default buffer that downstream parsers rely on. Assertion (per AC-HRN-CLS-008): unmarshal the JSONL line; confirm `len(member_keys) == 100`, `len(hamming_distances) == 20`, `truncated == true`, `hamming_pair_count == 4950`.

### EDGE-006 — Disk-full at cluster-merges.jsonl write time

If the OS reports `ENOSPC` when appending to `.moai/harness/cluster-merges.jsonl`, the audit log emission MUST log a non-blocking warning to stderr and continue. The merged Pattern MUST still be emitted to the in-memory pattern map (the audit log is best-effort, not blocking). Subsequent ClassifyTier and Promotion writes proceed.

### EDGE-007 — SimHash library load failure at startup

If the SimHash library (whether embedded or imported) fails to initialize (e.g., panic during `init()`, compile-time absence), the classifier MUST fall back to Stage-1-only mode per REQ-HRN-CLS-018 and emit a stderr warning. No panic MUST propagate to the caller.

### EDGE-008 — Mixed event types in Stage-1 singletons

Stage 2 MUST cluster singletons within the same `EventType` only. A `user_prompt` singleton MUST NOT cluster with a `subagent_stop` singleton even if their semantic features happen to produce low Hamming distance. The cluster grouping pre-filter MUST partition singletons by EventType before pairwise comparison.

### EDGE-009 — Concurrent AggregatePatterns invocations

If two callers invoke `AggregatePatterns` concurrently on the same log file, both MUST succeed (the file is read-only at this layer). Cluster-merges.jsonl appends use `O_APPEND|O_CREATE|O_WRONLY` semantics (atomic for ≤ PIPE_BUF lines on POSIX), so concurrent audit log writes MUST NOT interleave or corrupt. The pattern maps returned by each caller are independent (each call constructs its own map).

## Test Scenarios (Given-When-Then)

### Scenario 1: First-Time Operator Enables Stage 2

**Given** a project on V3R4-003 with `learning.classifier.stage_2_enabled: false` (default) and 200 days of accumulated `.moai/harness/usage-log.jsonl` containing 5000 singletons that have stalled at Tier-1 Observation,

**When** the operator edits `.moai/config/sections/harness.yaml` to set `learning.classifier.stage_2_enabled: true` and triggers the next batch learning cycle,

**Then** the classifier MUST run Stage 1 (preserving all 5000 Tier-1 records byte-identically per AC-HRN-CLS-001) AND Stage 2 (clustering similar singletons by SimHash).

**And** if 12 clusters of ≥ 3 members each form, MULTIPLE new merged Pattern records MUST appear in the aggregated map, of which some MAY classify to TierHeuristic, TierRule, or TierAutoUpdate.

**And** `.moai/harness/cluster-merges.jsonl` MUST contain 12 new JSONL lines (one per cluster) with member keys, Hamming distances, and merged metadata.

---

### Scenario 2: Privacy-Conscious User With Strategy A

**Given** a privacy-conscious user with `learning.user_prompt_content` ABSENT (Strategy A default from HARNESS-002 REQ-HRN-OBS-012) and `learning.classifier.stage_2_enabled: true`,

**When** the user submits 5 UserPromptSubmit events whose prompts contain sensitive PII (e.g., personal email, API keys),

**Then** the observer records only `prompt_hash`, `prompt_len`, `prompt_lang` per Strategy A (no `prompt_preview`, no `prompt_content`). Stage 2 fingerprints these singletons over their available semantic features `(subject, "", prompt_lang, "", "")` — note `prompt_preview` is absent so SimHash receives empty string for that field.

**And** the resulting cluster decisions MUST NOT contain any prompt plaintext. The cluster-merges.jsonl audit log MUST NOT contain the user's PII content.

**And** the static grep `grep "@example.com" .moai/harness/cluster-merges.jsonl` MUST return zero matches.

---

### Scenario 3: Power User With Strategy C Full Content

**Given** a power user with `learning.user_prompt_content: full` (Strategy C opt-in) and `learning.classifier.stage_2_enabled: true`,

**When** the user submits 5 UserPromptSubmit events with `PromptContent` populated,

**Then** Stage 2 MUST still NOT include `PromptContent` in the fingerprint or audit log (REQ-HRN-CLS-014). The classifier consumes only `PromptPreview` (which is also opt-in via Strategy B; if absent, the field is empty string).

**And** the fingerprints are computed over `(subject, prompt_preview, prompt_lang, agent_name, agent_type)` — `prompt_content` is structurally excluded by the feature-fields enumeration.

---

### Scenario 4: Operator Tunes Hamming Threshold

**Given** an operator who initially used the default `hamming_threshold: 3` and observed that genuinely-similar singletons in a noisy-feature project never cleared the tight threshold (zero merges over multiple batches),

**When** the operator loosens the threshold to `hamming_threshold: 5` (mid of research.md §8.3 recommended range 2-5),

**Then** the next batch run MUST use the new threshold. Some previously-isolated singletons MUST now cluster together (because their pairwise Hamming distance is between 3 and 5 — within the new looser threshold).

**And** existing merged Pattern records from previous batches remain in `.moai/harness/learning-history/tier-promotions.jsonl` unchanged (no retroactive un-merging; the audit log is append-only).

---

### Scenario 5: Operator Hits Rate Limit

**Given** an operator who enabled Stage 2 and within one week sees 12 Tier-4-eligible merged Patterns (some genuine, some false-positives),

**When** the orchestrator processes these 12 candidates,

**Then** only ONE `AskUserQuestion` invocation MUST occur within the 7-day window per REQ-HRN-FND-012 (preserved unchanged by REQ-HRN-CLS-015).

**And** the other 11 candidates MUST be deferred to subsequent weeks per the existing `safety/rate_limit.go` policy.

---

### Scenario 6: Invalid Config Drift

**Given** a developer who typo'd `learning.classifier.hamming_threshold: 3.5` (a float instead of int) during local config editing,

**When** the next batch learning cycle invokes `AggregatePatterns`,

**Then** the loader MUST detect the YAML type mismatch (int expected; float received) via `yaml.Unmarshal` error BEFORE `ClassifierConfig.Validate()` runs, log a one-line WARN to stderr identifying the malformed key, and fall back to `DefaultClassifierConfig()` (which uses `hamming_threshold: 3`). The classifier MUST then proceed in Stage-1-only mode per REQ-HRN-CLS-018 (because the fallback default has `stage_2_enabled: false`). No error MUST be raised; no clustering occurs; no audit log entry appears. (Covered by AC-HRN-CLS-014 sub-case 6.)

**And** the operator's existing pattern map remains valid; the only consequence is that Stage-2 merges are skipped for this batch.

---

### Scenario 7: Library Unavailable in Vendored Build

**Given** a vendored build of moai-adk-go where the SimHash library (`glaslos/go-simhash`) is missing from `vendor/` due to a `go mod vendor` mishap,

**When** the binary attempts to run Stage 2 at batch time,

**Then** the import resolution MUST fail at compile time (caught by CI). If the failure is at runtime (unlikely with Go's static linking but possible if dynamic loading is introduced), the classifier MUST fall back to Stage-1-only with a stderr warning.

**And** the embedded fallback SimHash implementation (if chosen by Wave B implementation per plan.md §3.2) MUST work without the external library, eliminating this risk entirely.
