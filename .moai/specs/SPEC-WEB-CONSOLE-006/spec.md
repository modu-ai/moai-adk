## §1 Context & Motivation

### 1.1 Where this SPEC sits in the cohort

The `web-console` line hardens and extends the loopback-only browser settings editor (`moai web`). The v3 cohort (001–005) is **completely closed** (all `completed`):

- **001** (모태) — loopback-only, no-auth profile/project settings editor.
- **002 / S1** — port `3041` default + web↔TUI validation parity (`model` / `effort_level` / `model_policy`).
- **003 / S2a** — flat project-config parity (`development_mode` + `git_convention.convention`).
- **004** — visual restyle (모두의AI design system: token layer + `console.css` component port + appbar + dark mode + theme toggle + inline-SVG icons + self-hosted Pretendard Latin+Hangul woff2 subset).
- **005 / S3** — interface i18n (en/ko/ja/zh `data-i18n` + appbar langpick + `localStorage`) + CJK self-host webfont coverage.

**006 / THIS SPEC** opens the **v4 cohort**. It is a *rendering-layer migration*: replace the `html/template` rendering engine + the vanilla form-submit flow with **Templ components + an HTMX foundation**, while preserving every observable behavior the v3 cohort established. It changes HOW the page is produced (Go codegen instead of runtime template parse) and lays the HTMX wiring, but it changes NOTHING the server contract or the user can observe. Section expansion — the deeply-nested config editing the v3 cohort kept deferring — is the *next* SPEC (007 / S2b); 006 builds the architecture that makes 007 tractable but implements none of its fields.

### 1.2 Why migrate (motivation)

The current renderer is a single ~339-line `internal/web/assets/page.html.tmpl` parsed at startup via `assets.go:pageTemplate()` (`html/template` + a hand-rolled `dict` map helper because `html/template` has no map constructor), with three inline `{{define}}` helpers (`langSelect`, `optSelect`, `icon`) and a 5th-fieldset structure that is hand-edited per field. Two structural pressures motivate the migration:

1. **Type safety + composability.** `html/template` directives (`{{template "optSelect" dict "Name" ... "Options" .DevelopmentModes}}`) are stringly-typed: a typo in a key, a missing `dict` pair, or a renamed view-model field surfaces only at render time. Templ produces **type-checked Go components** (`*_templ.go` generated by `templ generate`) — a field rename is a compile error, and a reusable `optSelect(...)` becomes a typed Go function. This composability is the prerequisite for 007's section-registry: 007 must add ~6 nested config sections, and hand-editing a ~339-line monolith per field does not scale.
2. **Progressive enhancement / partial-swap foundation.** The v3 console submits the whole form and re-renders the whole page. 007's nested-section editing wants *section-scoped* save + partial swap (edit one section without re-rendering the other 11). HTMX (`hx-post` returning an HTML partial swapped into a target) maps exactly onto Templ components (a partial = a Templ component rendered standalone). 006 embeds `htmx.min.js` and wires the form for progressive enhancement, establishing the foundation; 007 lands the section-scoped partial-swap.

The migration is **behavior-preserving by construction**: the existing test suite is the gate. If the migration changes any observable server behavior, a Class B test fails. If it changes the rendered markup contract, a Class A test fails. That is the safety net (§2).

### 1.5 The decision this SPEC resolves: HTMX usage depth in 006 (surfaced at GATE-2)

**Decision: 006 lands the Templ rendering migration + the HTMX *foundation* (embed `htmx.min.js`, wire the form for progressive enhancement) while keeping the `POST /save` observable response BEHAVIOR-IDENTICAL — full-page semantics preserved. The partial-swap section-navigation that genuinely exercises HTMX is 007 (S2b).**

Rationale (grounded in the characterization gate):

- **Full-page POST /save response is preserved.** Several Class B tests assert the POST /save response is a full page: `integration_test.go:TestGoldenPath_ReadWriteRoundTrip` reads back `value="Goos"` and `method="POST"` markers from the response body; `handlers_test.go:TestSaveValidRoundTrip` asserts `"Settings saved"` banner text in the full re-rendered page; the EC-2 atomic-reject tests assert the re-rendered form carries per-field errors. Introducing `hx-post` partial-swap on `/save` would change the response shape (a fragment instead of a full page) and break these tests. **006 therefore keeps `POST /save` returning a full page** (Templ-rendered instead of `html/template`-rendered, but the same full-document shape). The form MAY carry HTMX attributes for progressive enhancement, but the server's response to a normal POST is observably the same full page — the Class B integration/save tests stay green unmodified.
- **HTMX foundation that lands in 006:** (a) `htmx.min.js` embedded via `go:embed` + served at `/static/htmx.min.js` + linked from the page (offline, zero CDN); (b) the form MAY carry `hx-*` attributes for progressive enhancement that degrade gracefully (a non-JS / HTMX-disabled client still gets the full-page POST round-trip — the existing behavior). This establishes HTMX as available without changing the observable POST contract.
- **HTMX features deferred to 007 (S2b):** section-scoped `hx-post` partial-swap (save one section, swap only that section's fragment), `hx-target`/`hx-swap` section navigation, and any response-shape change to `/save`. These are exactly what nested-section editing needs and exactly what would break the full-page Class B assertions if done now.
- **Vanilla client-only behaviors stay vanilla.** The theme toggle (`localStorage("moai-console-theme")`) and the interface-language picker (`applyI18n` + `localStorage("moai-console-lang")`) are *client-only* preferences with no server round-trip. They do NOT fit HTMX's server-driven model and MUST NOT be forced into it. 006 preserves `app.js`'s theme + i18n + segment-visibility logic as-is; the Class B/A tests `TestInterfaceLanguageDoesNotAlterPOST`, `TestServerContractPreserved`, `TestLangAttrUpdatesOnSwitch`, `TestDarkModeAndThemeToggle` stay green.

This decision keeps the entire Class B server-contract suite green *unmodified* — which is the definition of "behavior preserving."

### 1.6 HTMX + Templ technical facts (validated via Context7 / official docs)

- **`templ generate` is a PURE GO tool** (`github.com/a-h/templ/cmd/templ`) — no Node, no npm, no Vite. It reads `*.templ` files and emits `*_templ.go` alongside them. The generated Go is compiled into the binary like any other Go source; it is NOT a runtime asset (so it does NOT need `go:embed` — only the static CSS/JS/font assets do). This is a genuine new build dependency (the project's `go.mod` has no `a-h/templ` today): the run-phase MUST install the `templ` binary, add a `go:generate` directive, and wire `templ generate` ahead of `go build` in the Makefile + CI (§ REQ-WC6-009/010/011).
- **Templ provides contextual auto-escaping** that replaces `html/template`'s escaping: `{ expr }` is HTML-escaped by default; `templ.Raw(...)` is reserved for known-safe markup. Security parity with `html/template` is preserved (no new injection surface).
- **`html/template` → Templ mapping** for the existing helpers: `{{range .Options}}<option .../>{{end}}` → Templ `for _, o := range options { <option .../> }`; `{{if index .Errors .Name}}has-error{{end}}` → `if errs[name] != "" { ... }`; `{{with index .FieldErrors .Name}}...{{end}}` → a typed `if msg := errs[name]; msg != "" { <span class="field-error">...{ msg }</span> }`; the `dict` helper disappears (Templ component args are typed Go parameters). The emitted markup (class names, attribute names/order-insensitive presence, `data-i18n` keys, `name=`/`id=` values) is preserved so the markup tests stay green.
- **HTMX self-host**: `htmx.min.js` (~14.5 KB, v2.x) is embedded via `go:embed` + served at `/static/htmx.min.js`. Zero CDN → the offline invariant is preserved.

### 1.7 Cohort scope-fence

006 is the v4 cohort's migration member. It does NOT add, remove, or re-scope any settings field; it does NOT change the server contract; it does NOT widen the v3 invariants (loopback bind, no-auth, host-check, server-canonical validation, persistence-via-seams, offline, section isolation, profile-vs-project scope). Section expansion (nested config editing) is **007 / S2b**, referenced only to delimit what 006 does NOT do (see §4). 006's Templ component registry + HTMX foundation is the architectural *enabler* for 007 — stated as rationale, implemented as zero S2b fields.

## §2 The Characterization Gate (the spine of this SPEC)

[HARD] The existing `internal/web` test suite (11 test files, ~70 functions) IS the behavior-preservation gate. The classification rests on **what each assertion OBSERVES**, NOT on the test's name:

- **Class B (server-contract SAFE)** — asserts observable server behavior via the HTTP layer (validation / persistence / HTTP status / host-check / section-isolation / scope, read from `httptest` responses or real-server round-trips). These define "동작 보존 (behavior preserving)" and MUST stay green **WITHOUT test-file modification**.
- **Class A (markup AT-RISK)** — asserts rendered-markup STRUCTURE (CSS classes / `data-*` / `name=` / `id=` / inline SVG, read from the rendered HTTP body) — the migration's risk surface; preserved via exact-markup parity, with §2.2-justified relaxations permitted.
- **Class C (implementation-coupled — NEW classification)** — reads an INTERNAL implementation artifact (the embedded `page.html.tmpl` source) OR calls a RETIRED internal symbol (`pageTemplate()`). These are NOT server-contract tests even when their *intent* is server-contract: their *current mechanism* reads source that 006 deletes / a symbol 006 retires, so they CANNOT stay unmodified and updating their mechanism (to assert against the new Templ render path / the rendered HTTP body instead) is EXPECTED and LEGITIMATE.

This A/B/C classification + porting strategy is the single most important deliverable of this plan; it defines what "behavior preserving" concretely means and where the migration risk lives.

## §3 GEARS Requirements

> Notation: GEARS (current). `<subject>` is generalized. Inherited HARD invariants (REQ-WC6-004…008, ported verbatim from 001–005) are Ubiquitous "shall" requirements; the migration mechanics are Event/State/Where requirements.

### REQ-WC6-001 (Ubiquitous — Class B server-contract tests stay green unmodified)

The migration **shall** keep every Class B server-contract test (§2.1: validation / persistence / HTTP routing / host-check / section-isolation / scope / golden-path) green **without modifying the test files**. The only permitted test-file change in Class B is the ADDITION of a new assertion for the HTMX foundation or build wiring (additive, never a modification of an existing Class B assertion). **When** any existing Class B assertion would require modification to pass, the migration **shall** halt and the agent **shall** return a blocker report, because that signals an out-of-scope observable-behavior change.

### REQ-WC6-002 (Ubiquitous — Class A markup tests preserved via exact-markup parity)

The Templ components **shall** emit markup that preserves the exact CSS class names, `data-i18n` / `data-theme` attributes, `name=` POST attributes, `id=` values, inline-SVG icon markup, and option-list iteration semantics, such that every Class A markup test (§2.2) stays green with minimal or no change. **Where** exact-markup parity is genuinely infeasible for a specific assertion, the migration **shall** relax that assertion to a semantic equivalent AND record a per-test justification in `acceptance.md` + `progress.md`. The migration **shall not** delete or wholesale-rewrite a Class A test to make it pass.

### REQ-WC6-003 (Ubiquitous — 5 fieldsets ported 1:1, zero field delta)

The Templ render **shall** reproduce the 5 existing fieldsets (Identity / Language / Launch / Statusline / Project) field-for-field, with every field's `name=` POST attribute, `id=`, option set, and order byte-identical to the pre-migration render, plus the non-fieldset chrome (appbar, hidden `__profile`, profile switcher, actions bar, FOUC `<head>` script, per-field error spans). The migration **shall not** add, remove, or re-scope any settings field or config section.

### REQ-WC6-004 (Ubiquitous — server-side canonical validation, no parallel rule set) [inherited 001/002/003]

The Console **shall** validate every submitted value server-side using the existing canonical predicates and lists in `validate.go` (`validatePrefs` + `validateProjectConfig`; `profile.IsValidPermissionMode`, `template.IsValidModelPolicy`, `models.DevelopmentMode().IsValid`, `config.IsValidConvention`; `langOptions` / `modelCanonical` / `effortLevelCanonical` / `statusline*Canonical` / `developmentModeCanonical` / `conventionCanonical`) and **shall not** introduce a parallel/duplicate rule set, a client-side validation SSOT, or any direct YAML marshal in the web layer. `validate.go` **shall** be byte-unchanged by 006.

### REQ-WC6-005 (Ubiquitous — persistence exclusively via existing seams) [inherited 001/003]

The Console **shall** persist exclusively through the existing injectable seams (`writePreferences` / `syncToProject` / `writeProjectConfig`, defaulting to `profile.WritePreferences` / `profile.SyncToProjectConfig` / the config-manager `writeProjectConfig`) and **shall not** perform a direct `yaml.Marshal` / `os.WriteFile` of any config section from the web layer. The migration **shall** preserve the `app` struct seams unchanged.

### REQ-WC6-006 (Ubiquitous — offline zero-network via go:embed, incl. htmx.min.js + Templ output) [inherited 004/005, extended]

The Console **shall** serve every frontend asset offline via `go:embed` with zero network fetch: `console.css`, `app.js`, `i18n.js`, the Pretendard + Noto CJK woff2 subsets + OFL licenses, AND the new `htmx.min.js`. The served CSS/HTML/JS **shall** contain zero external `https://` font/style/script URL (no Google-Fonts `@import`, no unpkg, no jsdelivr, no cdnjs). The Templ-generated `*_templ.go` is compiled Go (not an embedded data asset) and **shall not** require any runtime template/network fetch.

### REQ-WC6-007 (Ubiquitous — loopback-only bind + host-check, no auth/session/CSRF) [inherited 001]

The Console **shall** bind only to the loopback interface (`127.0.0.1`) and **shall** gate mutating requests (POST/PUT/PATCH) through the existing `hostCheckMiddleware` (loopback Host header required; GET ungated). The migration **shall** preserve `hostCheckMiddleware` and the loopback bind byte-for-byte and **shall not** add any auth, token, session, or CSRF infrastructure, nor a non-loopback bind.

### REQ-WC6-008 (Ubiquitous — section isolation + profile-vs-project scope) [inherited 001/003]

The Console persistence **shall** touch ONLY the in-scope sections (profile `preferences.yaml` via `WritePreferences`; user/language/statusline via `SyncToProjectConfig`; `quality.development_mode` + `git_convention.convention` via `writeProjectConfig`) and **shall not** write `workflow`, `harness`, `git-strategy`, `llm`, or any nested sub-field. The migration **shall** keep the `integration_test.go` DO_NOT_TOUCH sentinels (workflow.yaml / harness.yaml / git-strategy.yaml) intact, and **shall** keep profile fields scoped to `preferences.yaml` and project fields scoped to config sections.

### REQ-WC6-009 (Where — Templ is a first-class new build dependency)

**Where** the build produces the `internal/web` package, the build **shall** depend on `github.com/a-h/templ` (added to `go.mod`) and on the `templ` codegen tool (`github.com/a-h/templ/cmd/templ`), and `templ generate ./internal/web/...` **shall** run BEFORE `go build` / `go test` so the generated `*_templ.go` files exist for compilation. The migration **shall** pin the `templ` runtime library version in `go.mod` and document the codegen-tool install path.

### REQ-WC6-010 (Ubiquitous — go:generate directive + Makefile wiring)

The `internal/web` package **shall** carry a `//go:generate go run github.com/a-h/templ/cmd/templ generate` directive (or equivalent) so `go generate ./...` regenerates the `*_templ.go`, and the Makefile **shall** wire `templ generate` into the `generate` target AND ahead of the `build` / `test` targets so a clean checkout produces a buildable tree. The Makefile wiring **shall** be idempotent (re-running `templ generate` on unchanged `.templ` files is a no-op).

### REQ-WC6-011 (Ubiquitous — CI step + generated-file coexistence with go:embed)

The CI pipeline (and the local `scripts/ci-mirror/run.sh` mirror) **shall** run `templ generate` before `go build` / `go test`, so CI compiles the same generated tree. The generated `*_templ.go` files **shall** coexist with the existing `//go:embed` mechanism without conflict: the Templ output is compiled Go (NOT embedded as data); only the static assets (`console.css` / `app.js` / `i18n.js` / `htmx.min.js` / `fonts`) are embedded, and `assets.go`'s embed directive **shall** drop `page.html.tmpl` (no longer a runtime asset) and ADD `htmx.min.js`.

### REQ-WC6-012 (Ubiquitous — cross-platform build green)

The migrated package **shall** build on the host (`go build ./...` exit 0) AND cross-compile (`GOOS=windows GOARCH=amd64 go build ./...` exit 0) after `templ generate`, with no platform-specific Templ/HTMX dependency. The generated `*_templ.go` **shall** be platform-neutral pure Go.

### REQ-WC6-013 (Ubiquitous — closure gate)

The migration **shall** be considered complete only when `templ generate ./internal/web/...` succeeds AND `go test ./internal/web/... ./internal/config/... ./internal/cli/...` exits 0 (every Class B test green unmodified; every Class A test green with at most §2.2-justified parity adjustments) AND the host + Windows cross-builds exit 0.

### REQ-WC6-014 (Event-driven — POST /save full-page response shape preserved)

**When** the Console receives a `POST /save` (valid or atomic-reject), the Console **shall** return the same observable full-page response shape it returned before the migration (a complete HTML document, Templ-rendered instead of `html/template`-rendered), so the Class B golden-path / save / atomic-reject tests that assert full-page markers (`value=`, `method="POST"`, `"Settings saved"`, per-field errors) stay green. The migration **shall not** convert `/save` to an HTMX partial-swap response (that is 007 / S2b scope).

### REQ-WC6-015 (Where — HTMX foundation lands, partial-swap deferred)

**Where** the page is rendered, the Console **shall** embed `htmx.min.js` (self-hosted, `go:embed`, `/static/htmx.min.js`, linked from the page) and MAY decorate the form with `hx-*` attributes for progressive enhancement that degrade gracefully to the existing full-page POST round-trip when HTMX/JS is unavailable. The Console **shall not** implement section-scoped `hx-post` partial-swap, `hx-target`/`hx-swap` section navigation, or any `/save` response-shape change in 006 — these are deferred to 007 (S2b).

### REQ-WC6-016 (State-driven — client-only behaviors preserved as vanilla JS)

**While** the user toggles the theme or switches the interface language, the Console **shall** preserve the existing client-only vanilla-JS behavior (`localStorage("moai-console-theme")` theme toggle, `applyI18n` + `localStorage("moai-console-lang")` interface language, segment-visibility toggle) unchanged, and **shall not** route these client-only preferences through HTMX or any server round-trip. The interface language **shall** remain non-submitting (`uiLangSelect` carries no `name=`, sits outside `<form>`) so `TestInterfaceLanguageDoesNotAlterPOST` stays green.

### REQ-WC6-017 (Ubiquitous — Templ contextual auto-escaping security parity)

The Templ components **shall** rely on Templ's contextual auto-escaping for all dynamic values (`{ expr }` HTML-escaped by default), reserving `templ.Raw(...)` only for known-safe pre-escaped markup (the inline SVG icon set), preserving the `html/template` security posture with no new injection surface. The migration **shall not** emit any user-controlled value as raw unescaped HTML.

### REQ-WC6-018 (Ubiquitous — render-into-buffer error discipline preserved)

The Console **shall** render the Templ root component into a buffer first and, **when** rendering fails, surface a readable inline error (HTTP 500 with the cause) rather than a half-written 200 response — preserving the `handlers.go:render()` discipline (REQ-WC-010) and keeping `TestIndexReadErrorRendersInlineError` / `TestProjectReadSeamFailureRendersInlineError` green.

[HARD] The following are forbidden regardless of sibling-SPEC scope:

1. **No parallel/invented validation rule set** — validation stays in `validate.go` via the existing predicates/lists (REQ-WC6-004). 006 MUST NOT author a Templ-side or client-side validation rule set, MUST NOT touch `validate.go`. The server `.FieldErrors` re-render remains the validation SSOT.
