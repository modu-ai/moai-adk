## 0. Executive Summary

v2.14.0 is a targeted correctness release in the v2.x line. It closes the seven Critical-tier bugs identified in the utility review SYNTHESIS (MX validator, ast-grep integration, LSP subprocess) and lands three tightly-scoped SPECs — SPEC-UTIL-001/002/003 — whose acceptance criteria map one-to-one onto the audit citations (A1 §D1, A2 §D2-2, A3 §D2-3, A3 §D10-1). Each SPEC carries `breaking: false`; no user-visible API, config key, or wire format is altered. Additionally, v2.14.0 absorbs seven non-breaking subcommand fixes from B1/B2: the `team-reader` agent deletion regression in `team/plan.md` (B1 CF-1), the missing `/clear` directive in `plan.md` (B1 D1-P3), the Go-specific CI mirror drift in `sync.md` (B1 CF-5), the incorrect "/moai run" instruction in `sync.md` Phase 0.6 (B1 D4-P1), the `IsStagnantWithDiagnostics` dead code at `engine.go:64` (B2 P1-1), the missing no-op exit path in `fix`/`loop`/`codemaps` (B2 P1-2), and the three-way `max_iterations` drift between `ralph.yaml`/`loop.md`/`controller.go` (B2 P2-1).

What v2.14.0 deliberately does NOT deliver: the full type-unification refactor (Diagnostic × 3, Phase × 2, extension-map × 3) is deferred to v2.16 because it touches public hook JSON wire format; the `gopls_bridge` → `powernap_core` default flip is deferred to v3.0 because it is a breaking configuration change (BC-V3R2-UTIL-001); the Agentless codemaps refactor is deferred to v2.17 because it requires redesigning `codemaps.md` control flow away from the Explore subagent; and the v3R2 constitutional changes (38 SPECs, 16 breaking changes) remain on the v3 planning track per the user's strategic decision.

Non-breaking semver-minor compliance is enforced as a gate: every change that could be perceived as breaking is either restricted in scope or deferred. Estimated surface area: ~1,700 LOC modified across `internal/hook/mx/`, `internal/astgrep/`, `internal/lsp/core/`, `internal/lsp/subprocess/`, `internal/ralph/`, plus tree-sitter integration (~500 LOC for 16-language cyclomatic), 15 new ast-grep rule YAML files (5 languages × 3 rules), SPEC artifacts, and four-locale docs-site updates per CLAUDE.local.md §17.

## 0.5 Confirmed Design Decisions (2026-04-24)

Eight open questions were resolved via AskUserQuestion dialogs before SPEC writing began. Each decision is binding for v2.14.0 scope; changes require explicit user approval.

### D1 — SPEC naming convention
**Choice**: `SPEC-UTIL-001/002/003` (domain-only, no version prefix).
**Rationale**: version-agnostic. Same SPEC ID can carry continuing work into v2.15 without rename. v3R2 namespace (`SPEC-V3R2-UTIL-*` in planning track) is clearly distinguished by the `V3R2-` prefix.
**Impact**: SPEC directories created at `.moai/specs/SPEC-UTIL-001/`, `.moai/specs/SPEC-UTIL-002/`, `.moai/specs/SPEC-UTIL-003/`. Feature branches renamed to `feature/v2.14-util-001-*` pattern.

### D2 — Diagnostic type unification scope
**Choice**: Internal-only + Go type alias.
**Rationale**: `lsp.Diagnostic` becomes the single source. `hook.Diagnostic` and `gopls.Diagnostic` are declared as type aliases (`type Diagnostic = lsp.Diagnostic`). No change to hook JSON wire format, no breaking change for external consumers.
**Impact**: UTIL-003 scope includes adding type-alias declarations (~30 LOC across `internal/hook/`, `internal/lsp/gopls/`). Full unification (field consolidation) remains deferred to v2.16.
**Caveat**: Go type aliases are nominal identifiers — consumers accessing fields work; consumers using type assertions against the alias may need verification.

### D3 — LSP subprocess stderr policy
**Choice**: `io.Discard` drain goroutine.
**Rationale**: simplest pattern, deadlock eliminated, zero memory/disk footprint. Debugging insight trade-off acknowledged; richer stderr handling (file rotation, ring buffer, config-gated policy) deferred to v2.15 or later.
**Impact**: UTIL-003 adds single goroutine in `internal/lsp/core/client.go:213` post-`NewSupervisor`. ~5 LOC.

### D4 — Windows MX support approach
**Choice**: Go-native implementation (replace `grep` subprocess).
**Rationale**: full platform independence, Windows CI gate achievable, subprocess-spawn reduction (performance win), natural fix for unbounded-goroutine IMP-V3U-007. Implementation cost (~150 LOC) accepted.
**Impact**: UTIL-001 replaces `grep` subprocess calls with `filepath.WalkDir` + `bufio.Scanner` + `regexp`. `exec.Command("grep", ...)` invocations in `internal/hook/mx/validator.go:284, 317` removed. Bounded worker pool (IMP-V3U-007) emerges as a Go-idiomatic `sync.WaitGroup` + semaphore pattern.

### D5 — Cyclomatic complexity 16-language support
**Choice**: Tree-sitter Go bindings (direct AST query).
**Rationale**: moai FROZEN 16-language neutrality preserved. Subprocess-free (performance). McCabe complexity computed via tree-sitter queries per-language. Binary size cost (~10 MB per grammar × 16 = ~50-100 MB embedded) accepted in exchange for platform independence + multi-language.
**Impact**: UTIL-001 adds `github.com/smacker/go-tree-sitter` + 16 grammar sub-packages. New package `internal/hook/mx/complexity/` with per-language McCabe query definitions. ~500 LOC.
**Caveat**: tree-sitter node-type taxonomy differs per language (e.g., `if_statement` in Go vs `if_expression` in Rust). Per-language query files required. Initial v2.14 coverage: the 5 languages with existing ast-grep rules (Go/Python/TS/JS/Rust); remaining 11 queries scaffolded but return `complexity: 0` (documented limitation) until v2.15+ rule-seeding rotation.

### D6 — ast-grep suppression policy
**Choice**: Allow `// ast-grep-ignore` + require `// @MX:REASON <rationale>` on adjacent line.
**Rationale**: pragmatic flexibility + audit trail. Consistent with moai philosophy of requiring `@MX:REASON` pairing for all `@MX:WARN` / `@MX:ANCHOR` suppressions. CI lint rule validates pairing.
**Impact**: UTIL-002 adds lint rule checking that every `ast-grep-ignore` has a sibling `@MX:REASON` within 1 line. ~50 LOC CI check in `internal/hook/quality/astgrep_gate.go`. Violation produces `SUPPRESSION_WITHOUT_REASON` error with exit code 2.

### D7 — 11-language rule seeding timing
**Choice**: Seed 5 additional languages in v2.14 (Ruby/PHP/Elixir/C#/Kotlin); remaining 6 (Java/C++/Scala/R/Flutter/Swift) deferred to v2.15+ or community contribution.
**Rationale**: balances scope creep against 16-lang neutrality gap closure. v2.14 achieves 10/16 ast-grep language coverage — meaningful progress without overextending the bug-fix release. Each new language gets 3 foundational rules (unused-var, null-deref, todo-marker).
**Impact**: UTIL-002 creates 15 new YAML rule files under `.moai/config/astgrep-rules/{ruby,php,elixir,csharp,kotlin}/` (3 each). Each rule cites OWASP/CWE metadata per IMP-V3U-004 propagation. CI smoke test invokes `sg scan` against reference fixtures per language.

### D8 — Origin push timing
**Choice**: Immediate push on branch creation.
**Rationale**: transparency, CI availability, Git convention alignment. No quality-gate cost — push is not a merge; work continues on the branch.
**Impact**: executed 2026-04-24: `git push -u origin release/v2.14.0` completed; tracking relationship established; GitHub PR creation URL: `https://github.com/modu-ai/moai-adk/pull/new/release/v2.14.0`.

### Decision audit trail
All eight decisions recorded verbatim in AskUserQuestion dialog history (Batch 1: D1-D4, Batch 2 initial: D5-D8, Batch 2 revised: D5 re-asked after user clarification on 16-language requirement). The revised D5 supersedes the original gocyclo-only framing, which would have violated moai's FROZEN 16-language neutrality principle.

## 1. Strategic Context

### 1.1 Two-track model (user decision)

Track A — v2.x release line (production stability). One minor per rotation domain; each release lands non-breaking correctness work from the utility/subcommand audits plus the non-breaking subset of the paired v3R2 theme. Users stay on main and receive quality improvements without migration cost.

Track B — v3.0 planning (design explored, deferred). The 38 v3R2 SPECs, 16 documented breaking changes, and the constitutional reforms (CON-001..003, BC-V3R2-*) remain in `.moai/specs/SPEC-V3R2-*` and `docs/design/major-v3-master.md` on `main`. v3 planning artifacts are read-only references for v2.x work; they are not modified on the `release/v2.14.0` branch.

Rationale: the v3R2 bundle as a single jump is too large (38 SPECs + 16 breaking changes) to ship responsibly. Decomposing into five minor releases (v2.14 → v2.15 → v2.16 → v2.17 → v2.18) plus v3.0 gives users five incremental upgrade steps, each testable in isolation, with breaking changes concentrated at the v3.0 boundary.

### 1.2 Semver discipline

v2.14.0 MUST be backward compatible (semver minor). The three-way gate for "non-breaking":

- No user-visible API change (function signatures, CLI flags, hook JSON schema, SARIF field names).
- No user-visible config key change (`.moai/config/sections/*.yaml`).
- No user-visible wire format change (hook payloads, SARIF output, telemetry).

Any improvement that fails the gate is deferred to a minor that is paired with the v3R2 theme owning that type of change (v2.16 for types, v3.0 for breaking configs). Detection improvements (UTIL-001 method receiver) are considered non-breaking in the API sense even if they surface previously-hidden project-level violations — documented prominently in CHANGELOG.

### 1.3 Domain rotation schedule

| Minor | Domain | Core SPECs | v3R2 theme source |
|-------|--------|-----------|-------------------|
| v2.14 | Utility hardening | UTIL-001, UTIL-002, UTIL-003 | SYNTHESIS.md Tier 1 + B1/B2 audit |
| v2.15 | Runtime (non-breaking parts) | RT-004 typed session state, Hook JSON additive fields, structured logging | v3R2 RT-* (non-breaking subset) |
| v2.16 | Orchestration + Harness | ORC-004/005 (agent CI lint), HRN-001 (harness routing loader), effort calibration | v3R2 ORC-*, HRN-* |
| v2.17 | Workflow subcommands | WF-003 (multi-mode router), WF-004 (Agentless conformance), codemaps Agentless refactor | v3R2 WF-* |
| v2.18 | Extension + Migration | EXT-001..004 (plugin design, non-breaking), MIG memdir alignment | v3R2 EXT-*, MIG-* |
| v3.0.0 | Breaking changes + Constitution | CON-001..003, all BC-V3R2-*, type unification, gopls bridge retirement | Full v3R2 |

Each minor lands ~3-5 SPECs, avoiding the v3R2 big-bang risk profile.

## 2. v2.14.0 Scope (Tier 1 Critical only)

### 2.1 Critical bug fixes (from SYNTHESIS.md Tier 1)

All seven items below are classified P0 in SYNTHESIS.md Appendix Priority Score Matrix. Each is non-breaking (API-stable, wire-stable, detection-only improvement or internal-only fix).

1. **IMP-V3U-001 — MX method receiver blindspot** (SYNTHESIS §1.1 B1; `internal/hook/mx/validator.go:19`). `exportedFuncRe = ^func\s+([A-Z]\w+)` never matches `func (r *T) Method()`. Add `exportedMethodRe` matching `^func\s+\(\w+\s+\*?\w+\)\s+([A-Z]\w+)` and extend `extractFunctions` to run both regexes. Impact: method receivers — the majority of idiomatic Go exported API — become visible to P2/P3/P4 violation detection. Effort S. Non-breaking in API sense; may surface previously-invisible violations in user projects (documented in CHANGELOG §Detection Improvements).

2. **IMP-V3U-002 — LSP subprocess stderr leak** (SYNTHESIS §3.1 D2-3; `internal/lsp/core/client.go:213`). `result.Stderr` is never read; when a subprocess writes >64KB to stderr (gopls verbose logging), the pipe buffer fills and the subprocess blocks. Fix: `if result.Stderr != nil { go func() { defer result.Stderr.Close(); io.Copy(io.Discard, result.Stderr) }() }` after `NewSupervisor`. Effort XS. Purely internal.

3. **IMP-V3U-003 — LSP getOrSpawn concurrent client race** (SYNTHESIS §3.1 D10-1; `internal/lsp/core/manager.go:332`). Two goroutines race on the same language; both call `clientFactory`; a half-ready client enters the cache. Fix: add `sf singleflight.Group` field and gate client creation via `m.sf.Do(language, func() (any, error) { ... })`. Effort S. Purely internal; `golang.org/x/sync/singleflight` already transitively in `go.sum`.

4. **IMP-V3U-004 — ast-grep Rule metadata dropped** (SYNTHESIS §2.1 REC-01; `internal/astgrep/models.go:72`, `scanner.go:361`). YAML rules specify `metadata: {owasp: "A03:2021", cwe: "CWE-89"}` but the `Rule` struct has no `Metadata` or `Note` fields; all five security rules' OWASP/CWE classifications are silently dropped from SARIF output. Fix: add fields to `Rule`; propagate Rule→Finding in `scanWithRules`. Effort S. Additive to SARIF output — existing consumers see new fields, not renamed fields.

5. **IMP-V3U-005 — MX countFanIn substring false positives** (SYNTHESIS §1.1 B2; `internal/hook/mx/validator.go:264, 284`). `strings.Count(data, funcName)` and `grep -l funcName` match substrings; `funcName="New"` counts `NewContext`, `Renew`, `RenewToken`, inflating fan_in past the ANCHOR threshold for short names (`Get`, `Set`, `Parse`). Fix: grep `-w` flag; Go side `regexp.MustCompile("\\b" + regexp.QuoteMeta(funcName) + "\\b")`. Effort XS. Detection correctness; counts may decrease in existing projects (false-positive elimination).

6. **IMP-V3U-006 — ast-grep ScanMultiple unbounded goroutines** (SYNTHESIS §2.1 REC-02; `internal/hook/security/ast_grep.go:186`). 10K input files → 10K concurrent `sg` subprocesses. Fix: `sem := make(chan struct{}, runtime.NumCPU()*2)` bounded worker pool. Effort XS. Purely internal.

7. **IMP-V3U-007 — MX ValidateFiles worker pool** (SYNTHESIS §1.2 P1; `internal/hook/mx/validator.go:317`). Same unbounded-goroutine root cause as IMP-V3U-006 in a different subsystem. Fix: same semaphore pattern. Effort XS. Purely internal.

Acceptance criteria: integration tests show gopls verbose logging no longer deadlocks (>64KB stderr write completes without block); race detector on concurrent `getOrSpawn` shows single `clientFactory` call per language; type-alias compilation clean across hook/gopls consumers; no change to `hook` JSON wire format payload shape.

## 3. Subcommand Audit Findings Integration

### 3.1 Non-breaking fixes included in v2.14

**SC-4. sync.md Phase 0.6 instruction error** (B1 D4-P1). `sync.md` line 353 instructs users to "Run /moai run to add missing tags." The correct instruction is `/moai mx` — the dedicated MX workflow. **SC-5. Ralph Engine `IsStagnantWithDiagnostics` wiring** (B2 P1-1, engine.go:64). `loop.IsStagnantWithDiagnostics` exists, is tested in `stagnation_diag_test.go`, but is never called. **SC-6. No-op exit path for fix / loop / codemaps** (B2 P1-2, REQ-WF004-007). None of the three workflows implement explicit exit when localize finds zero targets. Current implicit behavior: continue through empty repair/validate phases. Fix: emit `status: no-op`, exit code 0, skip repair and validate. Exit code for edge case — previously undefined behavior.

**SC-7. max_iterations three-way drift consolidation** (B2 P2-1). `ralph.yaml:26` = 10; `loop.md:37` = 100; `controller.go:31` = 5. Fix: wire `ralph.yaml:loop.max_iterations` into `LoopController` via config loader; update `loop.md` documentation to reference config key instead of hardcoded 100; controller default becomes 10 (config-driven). Configuration consolidation — user-visible default unchanged for anyone reading `ralph.yaml`.
