> Strategy: 별도 트랙 유지 — v2.14 release track parallel to v3 planning track
> Branch: `release/v2.14.0` from `main`
> Date: 2026-04-24
> Scope: Tier 1 Critical bugs + 3 new utility SPECs + 7 non-breaking subcommand fixes
> Status: APPROVED (pending user execution of branch creation)

## 0. Executive Summary

v2.14.0 is a targeted correctness release in the v2.x line. It closes the seven Critical-tier bugs identified in the utility review SYNTHESIS (MX validator, ast-grep integration, LSP subprocess) and lands three tightly-scoped SPECs — SPEC-V2.14-UTIL-001/002/003 — whose acceptance criteria map one-to-one onto the audit citations (A1 §D1, A2 §D2-2, A3 §D2-3, A3 §D10-1). Each SPEC carries `breaking: false`; no user-visible API, config key, or wire format is altered. Additionally, v2.14.0 absorbs seven non-breaking subcommand fixes from B1/B2: the `team-reader` agent deletion regression in `team/plan.md` (B1 CF-1), the missing `/clear` directive in `plan.md` (B1 D1-P3), the Go-specific CI mirror drift in `sync.md` (B1 CF-5), the incorrect "/moai run" instruction in `sync.md` Phase 0.6 (B1 D4-P1), the `IsStagnantWithDiagnostics` dead code at `engine.go:64` (B2 P1-1), the missing no-op exit path in `fix`/`loop`/`codemaps` (B2 P1-2), and the three-way `max_iterations` drift between `ralph.yaml`/`loop.md`/`controller.go` (B2 P2-1).

What v2.14.0 deliberately does NOT deliver: the full type-unification refactor (Diagnostic × 3, Phase × 2, extension-map × 3) is deferred to v2.16 because it touches public hook JSON wire format; the `gopls_bridge` → `powernap_core` default flip is deferred to v3.0 because it is a breaking configuration change (BC-V3R2-UTIL-001); the Agentless codemaps refactor is deferred to v2.17 because it requires redesigning `codemaps.md` control flow away from the Explore subagent; and the v3R2 constitutional changes (38 SPECs, 16 breaking changes) remain on the v3 planning track per the user's strategic decision.

Non-breaking semver-minor compliance is enforced as a gate: every change that could be perceived as breaking is either restricted in scope or deferred. Estimated surface area: ~2,000 LOC modified across `internal/hook/mx/`, `internal/astgrep/`, `internal/lsp/core/`, `internal/lsp/subprocess/`, `internal/ralph/`, plus SPEC artifacts and four-locale docs-site updates per CLAUDE.local.md §17.

## 1. Strategic Context

### 1.1 Two-track model (user decision)

Track A — v2.x release line (production stability). One minor per rotation domain; each release lands non-breaking correctness work from the utility/subcommand audits plus the non-breaking subset of the paired v3R2 theme. Users stay on main and receive quality improvements without migration cost.

Track B — v3.0 planning (design explored, deferred). The 38 v3R2 SPECs, 16 documented breaking changes, and the constitutional reforms (CON-001..003, BC-V3R2-*) remain in `.moai/specs/SPEC-V3R2-*` and `docs/design/major-v3-master.md` on `main`. v3 planning artifacts are read-only references for v2.x work; they are not modified on the `release/v2.14.0` branch.

Rationale: the v3R2 bundle as a single jump is too large (38 SPECs + 16 breaking changes) to ship responsibly. Decomposing into five minor releases (v2.14 → v2.15 → v2.16 → v2.17 → v2.18) plus v3.0 gives users five incremental upgrade steps, each testable in isolation, with breaking changes concentrated at the v3.0 boundary.

### 1.2 Semver discipline

v2.14.0 MUST be backward compatible (semver minor). The three-way gate for "non-breaking":

- No user-visible API change (function signatures, CLI flags, hook JSON schema, SARIF field names).
- No user-visible config key change (`.moai/config/sections/*.yaml`).
- No user-visible wire format change (hook payloads, SARIF output, telemetry).

Any improvement that fails the gate is deferred to a minor that is paired with the v3R2 theme owning that type of change (v2.16 for types, v3.0 for breaking configs). Detection improvements (UTIL-001 method receiver) are considered non-breaking in the API sense even if they surface previously-hidden project-level violations — documented prominently in CHANGELOG.

### 1.3 Domain rotation schedule

| Minor | Domain | Core SPECs | v3R2 theme source |
|-------|--------|-----------|-------------------|
| v2.14 | Utility hardening | UTIL-001, UTIL-002, UTIL-003 | SYNTHESIS.md Tier 1 + B1/B2 audit |
| v2.15 | Runtime (non-breaking parts) | RT-004 typed session state, Hook JSON additive fields, structured logging | v3R2 RT-* (non-breaking subset) |
| v2.16 | Orchestration + Harness | ORC-004/005 (agent CI lint), HRN-001 (harness routing loader), effort calibration | v3R2 ORC-*, HRN-* |
| v2.17 | Workflow subcommands | WF-003 (multi-mode router), WF-004 (Agentless conformance), codemaps Agentless refactor | v3R2 WF-* |
| v2.18 | Extension + Migration | EXT-001..004 (plugin design, non-breaking), MIG memdir alignment | v3R2 EXT-*, MIG-* |
| v3.0.0 | Breaking changes + Constitution | CON-001..003, all BC-V3R2-*, type unification, gopls bridge retirement | Full v3R2 |

Each minor lands ~3-5 SPECs, avoiding the v3R2 big-bang risk profile.

## 2. v2.14.0 Scope (Tier 1 Critical only)

### 2.1 Critical bug fixes (from SYNTHESIS.md Tier 1)

All seven items below are classified P0 in SYNTHESIS.md Appendix Priority Score Matrix. Each is non-breaking (API-stable, wire-stable, detection-only improvement or internal-only fix).

1. **IMP-V3U-001 — MX method receiver blindspot** (SYNTHESIS §1.1 B1; `internal/hook/mx/validator.go:19`). `exportedFuncRe = ^func\s+([A-Z]\w+)` never matches `func (r *T) Method()`. Add `exportedMethodRe` matching `^func\s+\(\w+\s+\*?\w+\)\s+([A-Z]\w+)` and extend `extractFunctions` to run both regexes. Impact: method receivers — the majority of idiomatic Go exported API — become visible to P2/P3/P4 violation detection. Effort S. Non-breaking in API sense; may surface previously-invisible violations in user projects (documented in CHANGELOG §Detection Improvements).

2. **IMP-V3U-002 — LSP subprocess stderr leak** (SYNTHESIS §3.1 D2-3; `internal/lsp/core/client.go:213`). `result.Stderr` is never read; when a subprocess writes >64KB to stderr (gopls verbose logging), the pipe buffer fills and the subprocess blocks. Fix: `if result.Stderr != nil { go func() { defer result.Stderr.Close(); io.Copy(io.Discard, result.Stderr) }() }` after `NewSupervisor`. Effort XS. Purely internal.

3. **IMP-V3U-003 — LSP getOrSpawn concurrent client race** (SYNTHESIS §3.1 D10-1; `internal/lsp/core/manager.go:332`). Two goroutines race on the same language; both call `clientFactory`; a half-ready client enters the cache. Fix: add `sf singleflight.Group` field and gate client creation via `m.sf.Do(language, func() (any, error) { ... })`. Effort S. Purely internal; `golang.org/x/sync/singleflight` already transitively in `go.sum`.

4. **IMP-V3U-004 — ast-grep Rule metadata dropped** (SYNTHESIS §2.1 REC-01; `internal/astgrep/models.go:72`, `scanner.go:361`). YAML rules specify `metadata: {owasp: "A03:2021", cwe: "CWE-89"}` but the `Rule` struct has no `Metadata` or `Note` fields; all five security rules' OWASP/CWE classifications are silently dropped from SARIF output. Fix: add fields to `Rule`; propagate Rule→Finding in `scanWithRules`. Effort S. Additive to SARIF output — existing consumers see new fields, not renamed fields.

5. **IMP-V3U-005 — MX countFanIn substring false positives** (SYNTHESIS §1.1 B2; `internal/hook/mx/validator.go:264, 284`). `strings.Count(data, funcName)` and `grep -l funcName` match substrings; `funcName="New"` counts `NewContext`, `Renew`, `RenewToken`, inflating fan_in past the ANCHOR threshold for short names (`Get`, `Set`, `Parse`). Fix: grep `-w` flag; Go side `regexp.MustCompile("\\b" + regexp.QuoteMeta(funcName) + "\\b")`. Effort XS. Detection correctness; counts may decrease in existing projects (false-positive elimination).

6. **IMP-V3U-006 — ast-grep ScanMultiple unbounded goroutines** (SYNTHESIS §2.1 REC-02; `internal/hook/security/ast_grep.go:186`). 10K input files → 10K concurrent `sg` subprocesses. Fix: `sem := make(chan struct{}, runtime.NumCPU()*2)` bounded worker pool. Effort XS. Purely internal.

7. **IMP-V3U-007 — MX ValidateFiles worker pool** (SYNTHESIS §1.2 P1; `internal/hook/mx/validator.go:317`). Same unbounded-goroutine root cause as IMP-V3U-006 in a different subsystem. Fix: same semaphore pattern. Effort XS. Purely internal.

### 2.2 New SPECs (3)

Each SPEC is `lifecycle: spec-anchored` (maintained post-release for ongoing evolution), `breaking: false`, and follows the v3R2 10-section EARS format.

**SPEC-V2.14-UTIL-001 — MX Validator Correctness**. Owns IMP-V3U-001, 005, 007. Scope: `internal/hook/mx/validator.go` — method-receiver regex, word-boundary fan-in count, bounded worker pool. Acceptance criteria: characterization tests pin pre-fix behavior before changes; post-fix test suite demonstrates method receivers detected and substring false positives eliminated.

**SPEC-V2.14-UTIL-002 — ast-grep Integration Hardening**. Owns IMP-V3U-004, 006. Scope: `internal/astgrep/models.go`, `internal/astgrep/scanner.go`, `internal/hook/security/ast_grep.go` — Rule.Metadata/Rule.Note propagation, ScanMultiple semaphore, `detectSGVersion()` implementation (populates SARIF `tool.driver.version` from placeholder `"unknown"`). Acceptance criteria: SARIF output contains OWASP/CWE fields; concurrent scan bounded at `runtime.NumCPU()*2`; version field populated.

**SPEC-V2.14-UTIL-003 — LSP Subprocess Hygiene**. Owns IMP-V3U-002, 003. Scope (intentionally restricted for v2.14 non-breaking compliance): `internal/lsp/core/client.go` (stderr drain), `internal/lsp/core/manager.go` (singleflight spawn). Explicitly excluded from v2.14: Diagnostic type unification (defer to v2.16 as it touches hook JSON wire format), gopls bridge default flip (defer to v3.0 as BC-V3R2-UTIL-001). Acceptance criteria: integration tests show gopls verbose logging no longer deadlocks; race detector on concurrent `getOrSpawn` shows single `clientFactory` call per language.

### 2.3 Deferred items (rationale)

- **IMP-V3U-008 Diagnostic type unification** — deferred to v2.16 (types domain). The `hook.Diagnostic` JSON wire format (string severity) is external; migrating to `lsp.Diagnostic` (int severity) changes hook consumer payloads. Non-breaking requires a compatibility projection layer, which is v2.16 scope.
- **IMP-V3U-012/022 gopls bridge retirement + default flip** — deferred to v3.0 (breaking changes). `config.ResolveClientImpl()` default `"gopls_bridge"` → `"powernap_core"` is a breaking config change tracked as BC-V3R2-UTIL-001.
- **Tier 2 IMP-V3U-009/010/011/013/014/015/016/017/018/019/020/021** — deferred to v2.15/v2.16 per domain rotation. UTIL-017 (ValidationConfig wiring) and UTIL-018 (ast-grep context cancel leak) are candidates for v2.15 if runtime improvements warrant them.
- **Tier 3/4 strategic items (IMP-V3U-023..037)** — deferred to v3.0 planning per original SYNTHESIS §5 timeline (including LSPRAG adoption, `@MX:REASON` typed vocabulary, LSP 3.17 callHierarchy).

## 3. Subcommand Audit Findings Integration

### 3.1 Non-breaking fixes included in v2.14

Seven findings from B1/B2 fit the v2.14 non-breaking gate:

**SC-1. team/plan.md `team-reader` regression** (B1 CF-1, B1 D2-P1 / D7-P1). `team/plan.md` lines 60-108 spawn `Agent(subagent_type: "team-reader")`, but the `team-reader` agent was deleted in SPEC-TEAM-001. Team mode for the plan phase is currently broken. Fix: replace with `Agent(subagent_type: "general-purpose", model: "haiku", mode: "plan")` following the dynamic-team pattern. This is a regression fix, not a new feature — non-breaking.

**SC-2. plan.md `/clear` directive** (B1 D1-P3, CF-4). `spec-workflow.md` line 109 mandates `/clear` after plan completion; `plan.md` body has no `/clear` invocation. Only `team/plan.md` Phase 5 Cleanup executes it. Fix: add `/clear` to plan.md Completion section so sub-agent mode enforces the normative rule. Additive directive — non-breaking.

**SC-3. sync.md Go-specific CI mirror drift** (B1 CF-5, S1-P2). `sync.md` Phase 3.1.5 lines 823-845 hardcode Go-specific cross-compile commands targeting `./cmd/moai/`, in violation of CLAUDE.local.md §15 (template language neutrality). Fix: route via project language detection; each of the 16 supported languages gets equivalent test coverage, with Go's cross-compile matrix gated behind `language == "go"`. Structural cleanup — non-breaking.

**SC-4. sync.md Phase 0.6 instruction error** (B1 D4-P1). `sync.md` line 353 instructs users to "Run /moai run to add missing tags." The correct instruction is `/moai mx` — the dedicated MX workflow. Running `/moai run` from inside `/moai sync` creates a nested workflow invocation. Text fix — non-breaking.

**SC-5. Ralph Engine `IsStagnantWithDiagnostics` wiring** (B2 P1-1, engine.go:64). `loop.IsStagnantWithDiagnostics` exists, is tested in `stagnation_diag_test.go`, but is never called. The engine uses the weaker `loop.IsStagnant` (integer metrics only). Fix: replace single line in `engine.go:64`. Internal behavior improves (LSP diagnostic trends now influence convergence decisions); no external API change.

**SC-6. No-op exit path for fix / loop / codemaps** (B2 P1-2, REQ-WF004-007). None of the three workflows implement explicit exit when localize finds zero targets. Current implicit behavior: continue through empty repair/validate phases. Fix: emit `status: no-op`, exit code 0, skip repair and validate. Exit code for edge case — previously undefined behavior.

**SC-7. max_iterations three-way drift consolidation** (B2 P2-1). `ralph.yaml:26` = 10; `loop.md:37` = 100; `controller.go:31` = 5. Fix: wire `ralph.yaml:loop.max_iterations` into `LoopController` via config loader; update `loop.md` documentation to reference config key instead of hardcoded 100; controller default becomes 10 (config-driven). Configuration consolidation — user-visible default unchanged for anyone reading `ralph.yaml`.

- codemaps Agentless refactor (B2 P1-4) → v2.17 (workflow domain). Requires architectural redesign of `codemaps.md` Phase 1-3 control flow away from Explore/manager-docs subagents.
- manager-project delegation mismatch (B1 D2-P1) → v2.16. Either invoke `manager-project` in project.md Phase 3 or document it as CLI-only entry. Needs Agent Common Protocol decision.

- Agent Common Protocol scrub (9 agents) → v3.0 (breaking, per original master design).
- progress.md typed schema (B1 D3-P1, CF-3) → v2.15 (RT-004). SPEC-V3R2-RT-004 is already drafted; v2.15 lands the typed session state.
- Sprint Contract for `standard` harness (B1 R1-P1) → v2.16 (harness domain).
- `issue_number: 0 → null` (B1 D6-P1) → v2.15 with RT-004. Schema migration.

## 7. Branch Strategy

### 7.1 Branch creation

```bash
git checkout main
git pull origin main
git checkout -b release/v2.14.0
git push -u origin release/v2.14.0
```

### 7.2 Workflow

- All v2.14 work happens on `release/v2.14.0` branch via PR.
- PRs target `release/v2.14.0`, not `main`, for the duration of the release cycle.
- Feature branches for each SPEC and each subcommand fix group cut from `release/v2.14.0`:
  - `feature/v2.14-util-001-mx-validator`
  - `feature/v2.14-util-002-astgrep-hardening`
  - `feature/v2.14-util-003-lsp-hygiene`
  - `feature/v2.14-sc-skill-fixes`
  - `feature/v2.14-sc-sync-language-neutral`
  - `feature/v2.14-sc-ralph-engine-fixes`
- After `v2.14.0` tag: fast-forward merge `release/v2.14.0` → `main` (or rebase if main has moved forward with unrelated commits).
- Post-release: the `release/v2.14.0` branch stays alive for patch releases (v2.14.1, v2.14.2) if blocker regressions emerge in the field.

### 7.3 v3 track coexistence

- v3 SPECs remain in `.moai/specs/SPEC-V3R2-*` on `main`; not modified on `release/v2.14.0`.
- v3 research artifacts remain in `.moai/design/v3-research/` and `.moai/design/v3-redesign/` on `main`.
- v3 master design remains at `docs/design/major-v3-master.md` on `main`.
- No v3 work on `release/v2.14.0` branch; strictly backport-only if a v3 planning artifact is referenced from a v2.14 SPEC. Cherry-pick from main if a specific v3 doc is needed; otherwise reference by path on main.

### 7.4 Merge-back strategy

- v2.14.0 GA → tag + fast-forward merge `release/v2.14.0` → `main`.
- v3 track picks up v2.14.0 improvements via the main merge; any v3 SPEC that conflicts with v2.14 files is resolved in favor of the v2.14 version (non-breaking wins).
- v2.15 opens its own `release/v2.15.0` branch from the post-merge `main`.
