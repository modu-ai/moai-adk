# Korean Technical Writing Style Guide

## 문체 선택 원칙

**존댓말 vs 반말 기준**:

| 상황 | 권장 문체 | 이유 |
|------|----------|------|
| 입문서 (초급) | 존댓말 (합니다체) | 친근감, 독자 존중 |
| 레퍼런스/전문서 | 반말 (한다체) | 간결성, 전문성 |
| 튜토리얼/실습서 | 존댓말 (해요체) | 대화형, 친밀감 |
| API 문서 | 반말 (명사형) | 객관성, 정보 밀도 |

**문체 일관성 체크리스트**:
```
✓ 1장부터 마지막 장까지 동일한 문체 유지
✓ 본문과 예제 설명의 문체 통일
✓ 각주, 팁, 주의사항도 본문과 동일 문체
✓ 연습문제는 명령형(~하시오/~하세요) 허용
```

## 문장 구조 최적화

**명확성 원칙** (Kakao Enterprise 기술 문서 가이드 기반):
- **명확한 문장**: 핵심 용어와 문장을 모호하지 않게 사용하여 첫 독해에서 이해 가능
- **간결한 문장**: 꾸밈말이나 감탄사 없이 단순하고 쉬운 단어와 간결한 문장 사용

**Good 예시**:
```
이 알고리즘은 대규모 데이터셋에서 기존 방법보다 2배 빠른 성능을 보입니다.
현재 Netflix와 Spotify에서 프로덕션 환경에 적용 중입니다.
```

**문장 길이 가이드라인**:
- **최적**: 15-25어절 (읽기 편안함)
- **최대**: 35어절 (이상 시 문장 분리 권장)
- **단문 활용**: 중요한 개념 도입 시 10어절 이하 단문 사용

## 전문 용어 처리

**한글화 vs 영문 병기 기준**:

| 용어 유형 | 처리 방법 | 예시 |
|----------|----------|------|
| 널리 통용되는 외래어 | 한글 표기 | 알고리즘, 데이터베이스, 서버 |
| 최신 기술 용어 | 영문+한글 병기 (첫 등장) | 컨테이너(Container) |
| 약어 | 영문+풀네임 병기 (첫 등장) | API(Application Programming Interface) |
| 고유명사 | 원어 표기 | React, Python, AWS |
| 한국어 번역이 어색한 경우 | 영문 표기 | Refactoring (리팩토링보다) |

**주요 오류 표기 (절대 사용 금지)**:
- ❌ 어플리케이션 → ✅ 애플리케이션
- ❌ 아키텍쳐 → ✅ 아키텍처
- ❌ 캡쳐 → ✅ 캡처
- ❌ 챠트 → ✅ 차트
- ❌ 컨텐츠 → ✅ 콘텐츠

## 단락 구성 전략

**효과적인 단락 구조**:
1. **주제문** (Topic Sentence): 단락의 핵심 개념 제시
2. **지원 문장** (Supporting Sentences): 예시, 설명, 증거 제공
3. **결론/전환** (Concluding/Transition): 요약 또는 다음 단락 연결

**예시**:
```
[주제문] REST API는 상태를 저장하지 않는(stateless) 특성을 가집니다.

[지원문장] 각 요청은 독립적으로 처리되며, 서버는 이전 요청의 맥락을 기억하지 않습니다.
이로 인해 서버 확장이 용이하고, 장애 시 복구가 간단합니다.

[예시] 예를 들어, 로그인 상태는 서버가 아닌 클라이언트의 토큰으로 관리됩니다.

[전환] 이러한 무상태성은 캐싱 전략에도 영향을 미칩니다.
```
